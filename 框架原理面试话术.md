## 二、React 原理话术

### 2.1 Fiber 架构

**面试官问**："你了解 React Fiber 吗？"

**标准回答**：
```
"我深入学习过 Fiber 架构的设计思想。

【是什么】
Fiber 是 React 16 引入的新架构，它把渲染工作拆分成小的工作单元，
可以暂停、中断和恢复，解决了大型应用渲染卡顿的问题。

【为什么需要】
在 Fiber 之前，React 的协调过程是递归的，一旦开始就无法中断。
如果组件树很大，会长时间占用主线程，导致用户交互卡顿。

【怎么实现】
Fiber 把递归改成了链表结构，每个 Fiber 节点记录了 child（子节点）、
sibling（兄弟节点）、return（父节点）三个指针。

这样 React 可以在处理完一个 Fiber 节点后，判断是否有更高优先级
的任务（比如用户点击），如果有就先暂停，处理完再恢复。

【实际应用】
在我们的项目中，我利用 Fiber 的这个特性，使用 useTransition 
来标记低优先级更新（比如筛选结果渲染），让用户输入不会卡顿。"
```

**如果被追问细节**：
```
面试官："那你说说 Fiber 的双缓存机制？"

回答："Fiber 维护了两棵树，current 树和 workInProgress 树。
渲染时在 workInProgress 树上工作，完成后直接切换指针，
这样避免了中间状态暴露给用户。

这个机制在我学习时觉得很巧妙，类似于图形渲染的双缓冲技术。
不过具体的实现细节我没有深入到源码级别，主要是理解了它的
设计思想和应用场景。"

【重点：承认没看源码，但理解原理】
```

### 2.2 Hooks 原理

**面试官问**："说 Hooks 的实现原理"

**标准回答**：
```
"我深入研究过 Hooks 的工作机制。

【核心原理】
Hooks 是通过闭包和链表来实现的。React 内部为每个组件维护了
一个 Fiber 节点，节点上有个 memoizedState 属性，存储 Hooks 
链表。

每次调用 useState/useEffect，都会按顺序在链表上创建一个 Hook 
对象，所以 Hooks 必须在顶层调用，不能在条件语句里，否则链表
顺序会乱。

【useState 机制】
useState 返回的 setState 不是直接修改 state，而是创建一个更新
对象放入队列，然后触发调度。React 会在合适的时机（批量更新）
统一处理这些更新。

【useEffect 机制】
useEffect 的依赖数组，React 会在每次渲染时浅比较，判断是否
需要执行副作用。清理函数会在组件卸载或下次 effect 执行前调用。

【实际应用】
理解这些原理后，我在项目中更合理地使用 Hooks：
- 避免在循环或条件语句中使用
- 合理拆分自定义 Hooks，提高复用性
- 使用 useMemo/useCallback 优化性能时，明确依赖项"
```

**如果被追问**：
```
面试官："那 useState 的批量更新是怎么实现的？"

回答："React 18 之前，只有在事件处理函数中是批量更新，异步操作
（setTimeout/Promise）中是同步更新。

React 18 引入了自动批处理，所有更新都是批量的。底层是通过
调度器（Scheduler）来实现的，会收集同一时间段的所有更新，
统一处理。

具体的调度算法我没有深入到源码层面，但了解它使用了时间切片
和优先级队列的思想，这也是 Concurrent Mode 的基础。"
```

### 2.3 虚拟 DOM 与 Diff

**面试官问**："React 的 diff 算法原理"

**标准回答**：
```
"我学习过 React 的 diff 算法设计思想。

【为什么需要 diff】
直接操作真实 DOM 性能开销大，虚拟 DOM 通过对比找出最小差异，
再批量更新真实 DOM，提高性能。

【三大策略】
1. 同层比较：只比较同一层级，不跨层级（时间复杂度 O(n)）
2. 类型相同才复用：类型不同直接替换，不再比较子节点
3. 通过 key 优化：相同 key 的节点会尽量复用

【React 的 diff】
React 单向遍历新旧节点，通过 key 来判断节点是否可复用。
这也是为什么 React 强调 key 的重要性，key 不稳定会导致
不必要的重新渲染或状态丢失。

【实际应用】
理解 diff 算法后，我在写列表渲染时：
- 必定给 map 加上稳定的 key（用 id，不用 index）
- 避免不必要的层级嵌套
- 大列表用虚拟滚动，减少 diff 节点数"
```

## 三、Vue 原理话术

### 3.1 响应式原理

**面试官问**："Vue 的响应式原理是什么？"

**标准回答**：
```
"我深入学习过 Vue 2 和 Vue 3 的响应式机制。

【Vue 2 - Object.defineProperty】
Vue 2 使用 Object.defineProperty 劫持对象属性的 getter/setter。

初始化时，遍历 data 对象的所有属性，用 defineProperty 转换成
getter/setter。当访问属性时，收集依赖（Watcher）；当修改属性时，
通知所有依赖更新。

【Vue 2 的局限】
1. 无法检测对象属性的新增和删除（需要用 $set/$delete）
2. 无法检测数组索引和 length 的变化（重写了数组方法）
3. 需要递归遍历，初始化性能开销大

【Vue 3 - Proxy】
Vue 3 使用 Proxy 重写了响应式系统。

Proxy 可以拦截对象的所有操作（get/set/delete/has 等），天然
解决了 Vue 2 的局限性。而且 Proxy 是懒代理，只有访问到深层
属性时才会代理，性能更好。

【实际应用】
在我们的 Vue 项目中，我注意到：
- 避免在 data 中定义过深的嵌套对象（影响性能）
- 使用 Object.freeze 冻结不需要响应式的大数据
- 列表渲染时合理使用 key，避免不必要的更新"
```

### 3.2 虚拟 DOM 与 Diff

**面试官问**："Vue 的 diff 算法"

**标准回答**：
```
"我学习过 Vue 的 diff 算法原理。

【基本策略】
Vue 也是同层比较，时间复杂度 O(n)。

【Vue 2 的双端 diff】
Vue 2 使用双端比较算法：
- 新旧两个列表，各维护头尾四个指针
- 头头比、尾尾比、头尾比、尾头比
- 最大化复用节点，减少移动

这种算法在很多场景下性能更好，特别是列表反转、头尾插入等。

【Vue 3 的优化】
Vue 3 使用最长递增子序列算法，进一步优化了移动次数。

【实际应用】
理解 diff 算法后，我在写列表渲染时：
- 必定给 v-for 加上稳定的 key（用 id，不用 index）
- 避免不必要的层级嵌套
- 大列表用虚拟滚动，减少 diff 节点数"
```

## 四、状态管理原理话术

### 4.1 Redux 原理

**面试官问**："Redux 的工作原理是什么？"

**标准回答**：
```
"我深入学习过 Redux 的设计思想和核心原理。

【三大原则】
1. 单一数据源：整个应用的 state 存在一个 store 对象中
2. state 只读：只能通过 dispatch action 来修改
3. 纯函数更新：通过 reducer 纯函数返回新 state

【工作流程】
1. 组件 dispatch(action)
2. Redux 调用 reducer(prevState, action)
3. reducer 返回新的 state
4. Redux 更新 store，通知所有订阅者
5. 组件接收新 state，触发重新渲染

【核心实现】
- createStore：创建 store，内部维护 state 和 listeners
- dispatch：触发 action，调用 reducer，通知订阅者
- subscribe：订阅 state 变化

【中间件机制】
Redux 的中间件是通过 applyMiddleware 实现的，本质是对 dispatch
进行增强。常用的 redux-thunk 就是让 dispatch 可以接受函数。

【实际应用】
在我的 React 项目中，我用 Redux 管理全局状态，比如用户信息、
权限配置等。但对于组件局部状态，我会用 useState，避免过度
使用全局状态。

现在我更倾向于用 Zustand 这类轻量级状态库，API 更简洁，性能
也更好。"
```

### 4.2 Vuex 原理

**面试官问**："Vuex 和 Redux 有什么区别？"

**标准回答**：
```
"我在项目中同时用过 Vuex 和 Redux，对它们的设计思想有一些理解。

【核心差异】
1. 响应式 vs 发布订阅：
   Vuex 利用 Vue 的响应式系统，state 变化自动通知组件
   Redux 是发布订阅模式，需要手动 subscribe

2. 可变 vs 不可变：
   Vuex 的 mutation 可以直接修改 state（state.count++）
   Redux 必须返回新对象（return { ...state, count: state.count + 1 }）

3. 同步 vs 异步：
   Vuex 区分 mutation（同步）和 action（异步）
   Redux 通过中间件（redux-thunk）支持异步

【Vuex 的核心原理】
Vuex 的 store 是一个 Vue 实例，state 作为 data，所以天然具有
响应式。组件通过 computed 访问 state，自动收集依赖，state 变化
时组件自动更新。

【实际应用】
在我们的 Vue 项目中，我用 Vuex 管理：
- 用户信息、权限等全局状态（state + getters）
- 登录、退出等异步操作（actions）
- 状态修改统一通过 mutations（便于调试和追踪）

但也注意避免滥用，组件内部状态用 data，不是所有状态都放 Vuex。"
```

## 五、被追问细节的应对策略

### 策略1：承认边界，展示诚实

```
面试官："那你说说 React 的调度器具体是怎么实现的？"

回答："调度器的具体实现细节我确实没有深入到源码层面，我主要是
理解了它的核心思想：

1. 时间切片：把长任务拆分成 5ms 的小任务
2. 优先级队列：高优先级任务先执行
3. 可中断恢复：通过 Fiber 链表实现

如果入职后有机会，我很想深入研究这块的源码实现，了解底层的
调度算法。不过目前我更关注如何在业务中应用这些特性，比如用
useTransition 优化用户体验。"

【重点】
✅ 承认不懂细节（诚实）
✅ 说出核心思想（有基础）
✅ 表达学习意愿（态度好）
✅ 转到实际应用（展示能力）
```

### 策略2：转移到实践经验

```
面试官："那 Fiber 的 workLoop 是怎么实现的？"

回答："workLoop 的具体代码实现我没有研究过，但我知道它的工作
方式是通过 while 循环遍历 Fiber 链表，每处理完一个节点就判断
是否超时，超时就让出执行权。

在实际开发中，我更关注如何利用 Fiber 的能力优化性能。比如在
我们项目中，我用 React.memo 减少不必要的渲染，用 useTransition
标记低优先级更新，这些都是基于 Fiber 机制的应用。"
```
