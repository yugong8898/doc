## 一、评估结论总览

| 工具 / 概念 | 对 WLYD 适用性 | 推荐优先级 | 结论 |
|-------------|----------------|------------|------|
| Obsidian | 高 | 高 | 已适合直接配合 `.repoWiki/` 使用，提升知识检索、图谱和双向链接体验 |
| OpenSpec | 低到中 | 低 | 当前前端项目收益有限，除非后续做 BFF 或前后端契约治理 |
| spec-kit | 中 | 观察 | 适合需求规格、验收标准和 Agent 开发流程，但需结合现有 Prompts 体系评估 |

## 二、Obsidian

### 1. 是什么

Obsidian 是本地优先的 Markdown 知识管理应用，支持双向链接、标签、局部图谱、全局图谱和插件扩展。

### 2. 对 WLYD 的价值

WLYD 已有 `.repoWiki/` 作为统一知识库，结构包括 `README.md`、`kb/`、`maps/`、`handoff/` 等。Obsidian 可以直接打开 `.repoWiki/`，不需要改造目录结构。

主要价值：

- 通过双向链接查看文档引用关系。
- 通过知识图谱观察项目、模块、业务域之间的连接。
- 通过标签聚合坑点、需求、业务域。
- 通过 Dataview 自动查询 TODO、坑点、近期更新文档。
- 与 Cursor 配合使用时，可以先在 Obsidian 找到文档，再在 Cursor 中 `@` 引用。

### 3. 风险和边界

- Obsidian 是知识查看和整理工具，不替代 `.repoWiki/` 维护规则。
- `.repoWiki` 与代码冲突时仍以代码为准。
- Obsidian 个人工作区配置不应污染团队共享文档。

### 4. 结论

推荐继续使用。它是当前最小成本、最高收益的知识库增强方式。

## 三、OpenSpec

### 1. 是什么

OpenSpec 通常用于 API 或规格描述，容易与 OpenAPI、规格驱动开发、契约文档等概念关联。

### 2. 对 WLYD 的潜在价值

可能适用场景：

- 前后端接口契约治理。
- BFF 层接口定义。
- Mock Server 或联调契约校验。
- 复杂需求的输入、输出、验收标准固化。

### 3. 当前不推荐直接引入的原因

- WLYD 当前主要是多个前端项目，不是统一 API 服务端。
- 后端接口定义并不完全由前端维护。
- 现有 `.cursor/context/api-patterns.md`、`.repoWiki/kb/<project>/api.md` 已能覆盖前端接口调用知识。
- 引入 OpenSpec 需要后端、测试和产品协同，否则容易变成额外维护负担。

### 4. 结论

暂不作为近期落地重点。后续如果要做接口契约治理或 BFF，再重新评估。

## 四、spec-kit

### 1. 是什么

spec-kit 可理解为规格工具包，可能用于将需求、任务、验收标准、实现计划结构化，辅助 AI Agent 或团队按规格开发。

### 2. 对 WLYD 的潜在价值

WLYD 已有 `.md/prompts/` 角色体系、`.cursor/skills/` 工作流、`.cursorrules` 执行规范。spec-kit 类工具可能在以下方面有价值：

- 把需求拆解、方案、开发、审查、提测清单结构化。
- 让 AI Agent 按固定规格生成任务和验收标准。
- 补充现有 `wlyd-workflow`、`role-pm`、`role-architect`、`role-coder`、`role-reviewer` 的协作链路。

### 3. 风险和边界

- 不能替代现有 Prompts 和 Skills。
- 不能把未经确认的临时材料写入正式交付物。
- 如果格式过重，会降低日常开发效率。

### 4. 结论

作为观察项保留。后续如果发现能与现有 Prompt 体系融合，再做小范围试点。

## 五、Graphify / 图谱化工具

### 1. 是什么

Graphify 类工具通常用于把文档、代码、依赖关系或知识实体转换成图谱，用于关系分析和可视化。

### 2. 对 WLYD 的潜在价值

- 可视化组件、页面、接口之间的关系。
- 改动某组件时，辅助识别影响范围。
- 与 `.repoWiki/maps/`、`code-analyzer` Skill 形成互补。
- 对跨项目业务域，如结算、运单、对账，建立关系视图。

### 3. 当前不建议重型引入的原因

- 图数据库或复杂图谱平台维护成本较高。
- WLYD 当前更需要轻量、稳定、可人工校正的知识体系。
- Obsidian 图谱和静态依赖图已经能覆盖早期需求。

### 4. 结论

暂不引入重型 Graphify / Neo4j 类方案。先用 Obsidian 图谱和 madge 依赖图试点。

## 六、madge / 轻量依赖图

### 1. 是什么

madge 是一个 JavaScript / TypeScript 依赖分析工具，可以扫描文件依赖并生成图谱图片。

### 2. 对 WLYD 的价值

- 对 Vue2 项目的组件依赖、工具函数依赖有辅助分析价值。
- 可输出 SVG，放入 `maps/` 或 `docs/tech-radar/` 作为观察材料。
- 比 Graphify 更轻量，不需要图数据库。

### 3. 适用边界

- Vue 单文件组件的依赖解析可能需要结合实际构建配置验证。
- 动态 import、别名路径、运行时注册组件可能无法完全识别。
- 结果只能辅助判断，不能替代人工代码分析。

### 4. 结论

推荐作为后续轻量试点工具。先选 `logisticsweb` 中一个模块生成依赖图，不做全项目重型接入。

## 七、推荐落地顺序

1. 持续使用 Obsidian 增强 `.repoWiki/` 检索体验。
2. 建立技术雷达每日精选，持续观察 AI Coding、前端工程化和工具链信息。
3. 对 madge 做小范围试点，验证依赖图对影响范围分析是否有帮助。
4. 观察 spec-kit 类工具是否能融合现有 `.md/prompts/` 角色体系。
5. 暂缓 OpenSpec 和重型 Graphify，等待明确场景。
