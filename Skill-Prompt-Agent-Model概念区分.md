# Skill / Prompt / Agent / Model 概念区分

## 四个概念的关系

```
模型 (Model)          → 大脑，底层引擎，你选不了也改不了
  ↑ 驱动
Agent (智能体)        → 人，能读规则、用工具、调技能、自主完成多步任务
  ↑ 配备
Skill (技能)          → 背包里的工具，写好打包，用斜杠命令随时取用
  ↑ 简化版
Prompt (提示词)       → 便签纸，临时写的指令，用完就扔
```

## 逐个拆解

### 1. Model（模型）— 引擎

就是驱动 AI 的底层大语言模型。你不需要"使用"它，它是跑在背后的引擎。

- **你能做的**：在设置里选模型（如果支持）
- **你改不了的**：模型的能力边界、推理方式
- **项目中的例子**：Qoder 背后跑的那个模型，就是引擎

### 2. Prompt（提示词）— 便签纸

临时写的一段指令，告诉 AI 这次要干什么。

- **特点**：一次性、临时性、每次手写
- **项目中的例子**：`.md/prompts/role-coder.md` 就是 prompt，在对话里 `@role-coder.md` 引用它
- **缺点**：每次都要手动引用，容易忘，格式不统一

### 3. Skill（技能）— 工具箱

把 prompt **打包升级**：加上名称、描述、触发条件，注册成可复用的技能，用斜杠命令激活。

| 对比 | Prompt | Skill |
|------|--------|-------|
| 激活方式 | 手动 `@文件路径` | `/skill-name` 一键调用 |
| 发现性 | 靠记忆，容易忘 | 系统列出可用技能 |
| 复用性 | 每次引用 | 写一次，到处调 |
| 结构 | 纯 markdown | markdown + frontmatter 元数据 |

- **项目中的例子**：`.qoder/skills/commit-message/SKILL.md` 就是 Skill，输入 `/commit-message` 就能用
- **适合场景**：反复做的标准流程（提交信息、代码审查、需求分析）

### 4. Agent（智能体）— 执行者

能**自主决策**的实体，可以读规则、用工具、调技能、多步推理完成复杂任务。

- **主 Agent**：对话中的 AI 助手，能读 `.cursorrules`、能搜索代码、能编辑文件
- **Sub Agent**：专用子智能体，比如 Qoder 内置的 `CodeReview`（代码审查）、`Debug`（调试）
- **项目中的例子**：你说"帮我改这个需求"，Agent 会自动读规则 → 搜索代码 → 分析影响 → 写代码 → 自检，这一串是 Agent 自主完成的

## 一句话记忆法

| 概念 | 一句话 | 项目中的例子 |
|------|--------|--------------|
| **Model** | 跑在背后的引擎，不用管 | Qoder 的大模型 |
| **Prompt** | 临时便签，用完即弃 | `@role-coder.md` |
| **Skill** | 打包好的工具，`/命令` 调用 | `/commit-message` |
| **Agent** | 干活的人，能读规则用工具 | 对话中的 AI 助手 |

## 项目中怎么搭配用

```
Agent（AI 助手）
├── 自动读取 → .cursorrules / .qoder/rules/（项目规则，每次生效）
├── 手动引用 → .md/prompts/role-xxx.md（Prompt，按需引用）
├── 斜杠命令 → .qoder/skills/xxx（Skill，按需触发）
└── 派子智能体 → CodeReview / Debug（Sub Agent，专用任务）
```

## 最佳实践

- **规则类**（每次都要遵守的）→ 放 `.cursorrules` / `.qoder/rules/`
- **流程类**（反复执行的标准步骤）→ 转成 Skill
- **临时类**（偶尔用一次的）→ 直接 Prompt 就行，不用转 Skill
- **复杂类**（需要多步自主完成的）→ 交给 Agent，必要时派 Sub Agent
