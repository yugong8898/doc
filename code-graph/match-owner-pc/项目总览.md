# match-owner-pc 项目总览

## 概述

`match-owner-pc` 是货主端 PC 侧的前端工程，基于 **qiankun 微前端架构**构建，由一个主应用（`app-main`）和多个子应用（`app-core`、`app-base`、`app-dispatch`）组成。项目使用 Vue 3 + TypeScript + Element Plus + Pinia 技术栈，构建工具为 Vite。

## 文件结构

```
fe-match-end/match-owner-pc/
├── app-main/          # 主应用（qiankun 宿主）
├── app-core/          # 子应用：商品交易（货源、运力、订单）
├── app-base/          # 子应用：基础配置（运单、财务、系统设置等）
└── app-dispatch/      # 子应用：调度中心
```

## 应用拆分与职责

| 应用 | 名称 | 路由前缀 | 端口 | 核心职责 |
|------|------|---------|------|---------|
| `app-main` | 主框架 | `/` | - | 登录、布局、菜单、权限、微前端编排 |
| `app-core` | 商品交易 | `/app-core` | 5679 | 货源发布、运力管理、订单管理、司机/车辆管理 |
| `app-base` | 基础配置 | `/app-base` | 5680 | 运单管理、财务管理、系统设置、报表中心 |
| `app-dispatch` | 调度中心 | `/app-dispatch` | 5681 | 发货单调度、智能调度 |

## 核心逻辑

### 1. 微前端架构（qiankun）

**主应用注册子应用**，在 `app-main/src/config/micro-app-config.ts` 中集中维护所有子应用配置：

```typescript
const MICRO_APPS = {
  core:     { name: 'app-core',     activeRule: '/app-core',     entry: `${origin}/ng-app-core/` },
  base:     { name: 'app-base',     activeRule: '/app-base',     entry: `${origin}/ng-app-base/` },
  dispatch: { name: 'app-dispatch', activeRule: '/app-dispatch', entry: `${origin}/ng-app-dispatch/` },
  '3pl':    { name: 'app-3pl',      activeRule: '/app-3pl',      entry: `${VITE_3PL_URL}/yw/` },
}
```

**主应用向子应用注入 props**（`app-main/src/micro-apps.ts`）：

- `routerBase` — 子应用路由基础路径
- `mainRouter` — 主应用 Router 实例
- `getToken` / `getCookieToken` — 认证 token 获取方法
- `userStore` — 用户状态（Pinia store）
- `organizationStore` — 组织状态
- `routes` — 动态路由（getter，确保每次挂载读取最新路由）
- `actions` — 主应用暴露的能力（刷新企业列表、账号切换、菜单权限判断）

### 2. 应用启动流程

`app-main/src/main.ts` 中的启动链路：

```
asyncInitDeviceId()
  → initTrack()（埋点初始化）
  → getPlatformConfig()（拉取平台配置）
    → 处理 URL ticket 参数（授权登录场景）
    → getToken() 检查登录状态
    → userStore.fetchUserInfo()（拉取用户信息）
    → userStore.switchToCompanyAccount()（个人账号自动切换企业）
    → app.mount('#app')
```

**授权登录（SSO）特殊流程**：URL 中携带 `ticket` 时，走 `handleAuthorizationLogin()` 处理，成功后用 `history.replaceState` 清除 ticket 参数（避免 qiankun 跨域刷新问题）。

### 3. 用户状态管理

`app-main/src/store/user/index.ts`（Pinia store `user-info`）管理：

- 用户信息（`userInfoModel`）：`mtUserDTO`、`memberDTO`、`companyDTO`、`orchTokenInfoDTO` 等
- 账号类型判断：个人账号（10）vs 员工/企业账号（20）
- 企业切换逻辑（`switchToCompanyAccount`）：登录/刷新时自动切换默认企业
- 营业资质过期管控（`checkBusinessLicenseExpired`）
- 国企/央企标识（`checkCarrierType`）

**账号切换流程**（`switchToCompanyAccount`）：
```
getMyCompanyList()
  → 查找 loginCompany / defaultCompany
  → switchCompany({ targetSysType, loginId, token })
  → 拿到 targetToken → handleSwitchSuccess()
    → setToken + fetchUserInfo + clearAllCachePage + initRouter
```

### 4. 路由系统

- 静态路由：`app-main/src/router/modules/` 目录自动导入
- 动态路由：登录后通过 `initRouter()` 从后端获取，注册到 Vue Router
- 路由状态管理：`routerStateManager`（`router-state-manager.ts`）管理路由加载状态，避免微前端场景下重复初始化
- 导航守卫：`beforeEach` 处理权限拦截、营业资质过期拦截、动态路由初始化、多标签页管理

### 5. 子应用路由加载策略

子应用通过 `vite-plugin-qiankun` 集成，路由配置通过主应用 props 注入（`routes` getter）。子应用启动时无需自行拉取路由，由主应用的 `routerStateManager` 统一管理并注入。

## 注意事项

1. **子应用端口固定**：`app-core: 5679`、`app-base: 5680`、`app-dispatch: 5681`，本地开发需要各自单独启动。
2. **本地调试子应用**：在 `app-main/.env.dev` 中设置 `VITE_LOCAL_MICRO=true` 后，主应用会使用子应用的本地端口地址（`localEntry`）。
3. **路由前缀隔离**：所有子应用路由必须以对应的 `activeRule` 前缀开头，避免路由冲突。
4. **账号切换不能整页刷新**：在 `switchToCompanyAccount`（登录/刷新流程）中，切换成功后只能清缓存 + `initRouter`，不能用 `window.location.href` 刷新，否则会触发无限循环（qiankun 跨域问题）。
5. **Element Plus 命名空间**：主应用使用 `namespace: 'app-main-el'`，子应用各自独立，避免样式冲突。
6. **`app-core` 路由文件为空**：`app-core/src/router/routes.ts` 导出空数组，路由完全由后端动态下发，通过主应用 props 注入。
