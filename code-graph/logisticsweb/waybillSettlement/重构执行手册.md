## 第一波：修 Bug（用户直接受影响）

### T1 — getDetails 接口报错后页面永久 loading

**文件**：`edit.vue`

**现状**：`getDetails` 方法用 `this.loading` 做防重，开头置 true，但只在 `if (datas)` 成功分支内重置为 false。catch 分支没有重置，导致接口报错后 loading 永远为 true，页面卡死。

**目标**：无论接口成功还是失败，loading 都能正确重置。用户遇到网络异常时页面可重试，不再需要关闭标签页。

**现象**：用户打开调整结算页时，如果详情接口 500/超时，页面一直转圈无法操作，刷新也没用（loading 标志位不重置，防重逻辑阻止再次请求）。

**根因**：`loading` 只在 `if (datas)` 成功分支里重置为 false，catch 分支遗漏。

**改法**：
```js
async getDetails() {
  if (this.loading) return
  this.loading = true
  try {
    // ... 原有逻辑不动
  } catch (e) {
    // 原有错误处理
  } finally {
    this.loading = false  // ← 唯一新增
  }
}
```

**测试重点**：
| 场景 | 预期 |
|------|------|
| 正常进入编辑页 | 数据正常加载，loading 消失 |
| 接口 500（DevTools Network 里 block 请求模拟） | 页面展示错误提示，loading 消失，可重试 |
| 快速双击进入编辑页 | 不发重复请求 |

**收益**：修复线上真实 bug，用户不再遇到"白屏卡死只能关页面"的情况。

### T2 — getRiskDetails 风控数据串页

**文件**：`edit.vue`

**现状**：`getRiskDetails` 方法直接将查询结果写入 `this.highMidWaybillListPage`，但在发起新请求前不清空旧数据。当新请求返回空结果时，页面仍展示上一张运单的风控数据。

**目标**：每次进入新运单时，风控面板只展示当前运单的数据。无数据时显示为空，不残留任何历史记录。

**现象**：从一张中高风险运单返回列表，再进入一张低风险运单，风控面板残留上一张的风控结果。

**根因**：`getRiskDetails` 发请求前没有清空 `highMidWaybillListPage`，新请求返回空时旧数据仍在。

**改法**：
```js
async getRiskDetails() {
  this.highMidWaybillListPage = []  // ← 唯一新增
  // ... 原有查询逻辑不动
}
```

**测试重点**：
| 场景 | 预期 |
|------|------|
| 中高风险运单 | 风控面板展示当前运单风控结果 |
| 从中高风险返回 → 进入低风险运单 | 风控面板为空 |
| 接口报错 | 风控面板为空，不残留旧数据 |

**收益**：消除风控数据错误展示，避免运营误判风险等级。

## 第二波：费用明细修正（金额计算准确性）

### T3 — countMoney 硬编码下标导致 B1 企业金额写错行

**文件**：`CostMessageDetails.vue`

**现状**：`countMoney` 方法通过 `this.newList[0]` 硬编码取第一行来写入运费差价金额。普通运单下 `newList[0]` 确实是运费差价行，但 `companyType=1`（B1 企业）时 `newList[0]` 是货物保障服务行，导致金额写到了错误的位置。

**目标**：`countMoney` 通过 `costName` 查找目标行，不依赖数组下标。无论 `newList` 的顺序如何变化，运费差价行的金额都能正确写入；找不到目标行时安全跳过，不报错。

**现象**：`companyType=1`（B1 企业）时，`newList[0]` 是"货物保障服务"行而非"运费差价"行，`countMoney` 把计算结果写到了错误的行上。普通运单不受影响。

**根因**：`this.$set(this.newList[0], 'confirmMoney', serviceRateMoney)` 用硬编码下标 `[0]` 取目标行。

**改法**：
```js
// 改前
this.$set(this.newList[0], 'confirmMoney', serviceRateMoney)

// 改后
const freightDiffRow = this.newList.find(i => i.costName === '运费差价')
if (freightDiffRow) {
  this.$set(freightDiffRow, 'confirmMoney', serviceRateMoney)
}
```

**测试重点**：
| 场景 | 预期 |
|------|------|
| 普通运单（companyType≠1） | 修改确认金额 → 运费差价行联动更新 |
| B1 企业运单（companyType=1） | newList 无运费差价行时不报错、不乱写 |
| 连续多次修改确认金额 | 每次联动结果正确 |

**边界情况**：
- `newList` 为空数组 → find 返回 undefined → if 兜底不执行，不崩溃
- 运费差价行存在但 `costName` 被后端改名 → 需确认字段值是否稳定

**收益**：修复 B1 企业场景下费用金额计算错误，直接影响结算准确性。

### T4 — CostMessageDetails 双重初始化消除

**文件**：`CostMessageDetails.vue`

**现状**：`newList`（费用明细特殊行）有两个初始化入口——`mounted` 钩子里初始化一次（约 50 行），`detailObj` watch 里再初始化一次（逻辑几乎相同但细节不一致）。mounted 的结果是"过渡态"，几乎立刻被 watch 覆盖。两处在全流程运单时对 `premiumAmount` 的处理不一致（mounted 不强制为 0，watch 会强制），新增业务逻辑时不知道该改哪一处。

**目标**：`newList` 只有一个初始化入口（`detailObj` watch），逻辑集中在 `_buildNewList` 和 `_updateOilGasOnly` 两个命名清晰的方法里。后续修改费用明细初始化逻辑只需改一处，行为完全可预测。

**现象**：`mounted` 和 `detailObj` watch 各初始化一次 `newList`，mounted 的结果立刻被 watch 覆盖。两处逻辑存在细微差异——全流程运单时 mounted 不强制 `premiumAmount=0`，watch 会强制——导致极短时间内展示数据闪烁且不一致。

**根因**：两份逻辑重复维护，随业务迭代产生了不一致。

**改法**：
1. 删除 `mounted` 里约 50 行 `newList` 初始化代码
2. `detailObj` watch 作为唯一入口
3. 提取两个方法明确职责：

```js
watch: {
  detailObj(val) {
    this.$nextTick(() => {
      if (this.newList.length === 0 || this.type !== 'edit') {
        this._buildNewList(val)       // 完整重建
      } else {
        this._updateOilGasOnly(val)   // 编辑模式：只更新油气行
      }
    })
  }
}
```

**测试重点**：
| 场景 | 预期 |
|------|------|
| 进入 edit 页面 | 运费差价/货物保障/油气优惠三行正常展示 |
| 全流程运单 | 货物保障服务金额显示 0 |
| companyType=1（B1） | 只有货物保障服务行，无运费差价行 |
| 保存后刷新 | newList 正确重建，不残留旧值 |
| 查看模式（read） | 费用明细行展示正确，金额不可编辑 |

**边界情况**：
- `detailObj` 初始值为空对象时 watch 不应执行构建 → `_buildNewList` 内部需判断 `val` 有效性
- 编辑模式下切换运单（不关页面直接改路由参数） → `newList.length > 0` 但数据是旧的 → 需要用 `busBillId` 变化来判断是否完整重建

**收益**：
- 减少约 50 行重复代码
- 消除全流程运单 premiumAmount 展示不一致的潜在 bug
- `newList` 有唯一数据来源，后续维护时不需要"两处都改"

## 第三波：提交流程重构（开发体验 + 修 3 个隐性 bug）

### T5 — 保存/提交 5 级调用链拍平

**文件**：`edit.vue`

**现状**：保存/提交流程是 5 级嵌套调用链 `handleSave → changeWaybillStatus → beforeChangeWaybill → checkOrder → riskManagement → postSubmitAfterRiskPass`，每级有副作用且异步风格混用（.then 和 async/await 交替）。`btnLoading` 的重置分散在不同方法里，新增校验步骤需要通读 200+ 行代码才能找到插入点。同时存在 3 个隐性 bug（抹零重复计入、死代码防重、失败原因不展示）。

**目标**：保存/提交统一为一个线性 `_executeFlow(action)` 方法，每个校验步骤是独立的 async 方法（返回 boolean），步骤之间无嵌套关系。新增/删除校验只需在 `_executeFlow` 里加减一行。`btnLoading` 统一在 finally 重置。3 个隐性 bug 同步修复。

**现象（开发痛点）**：
- 新增一个前置校验（比如"结算重量不能为0"），找不到该插在哪
- 追一个提交失败的 bug，要在 5 个方法之间跳转：`handleSave → changeWaybillStatus → beforeChangeWaybill → checkOrder → riskManagement → postSubmitAfterRiskPass`
- `.then()` 和 `async/await` 混用，`btnLoading` 重置时机分散在不同方法里

**同时修复的 3 个隐性 bug**：

| # | bug 描述 | 影响 |
|---|---------|------|
| 1 | 抹零弹窗取消后，`costNewList` 里已 push 的抹零行没清理，下次提交重复计入 | 费用多算 |
| 2 | `riskManagement` 开头 `if (this.btnLoading) return` 是死代码（进来前 loading 已被置 false），永远不触发 | 无实际防重能力 |
| 3 | `handleSubmitRiskDialog` 里 `riskResult = null` 从未赋值，失败原因永远走通用分支，详细原因不展示 | 运营看不到具体失败原因 |

**改法**：将 5 级调用链拍平为一个线性流程：

```js
handleSave()   → _executeFlow('save')
handleSubmit() → _executeFlow('submit')

async _executeFlow(action) {
  this.btnLoading = true
  try {
    if (!await this._validateCostDetails()) return   // 费用明细校验
    if (!this._validateCargoLoss()) return            // 货损确认校验
    if (!this._validateOilGasCost()) return           // 油气费不超运费
    if (!await this._handleZeroMode()) return         // 抹零处理（取消时内部清理）
    if (!await this._validateForm()) return           // 表单字段校验
    if (!await this._checkOrderStatus()) return       // 运单状态接口校验
    await this._runRiskAndSubmit(action)              // 预保存 → 风控 → 正式提交
  } finally {
    this.btnLoading = false
  }
}
```

**关键设计保留**：`_runRiskAndSubmit` 内必须保留"预保存（当前状态）→ 风控测算 → 正式提交（320）"的两次接口顺序。风控接口依赖后端已固化的费用数据，不能合并为一次调用。

**抹零取消的修复**：
```js
async _handleZeroMode() {
  const diff = this._calcDiffAmount()
  if (!this.zeroByBusIdMode || diff === 0) return true
  this._appendZeroRow(diff)
  const confirmed = await this._showZeroConfirmDialog(diff)
  if (!confirmed) {
    this._removeZeroRow()  // ← 关键：取消时清理已追加的行
    return false
  }
  return true
}
```

**测试重点**：
| 场景 | 预期 |
|------|------|
| 保存（310） | 费用校验 → 油气校验 → 表单校验 → 接口成功 → 跳回列表 |
| 提交（320） | 全部校验 → 预保存 → 风控 → 正式提交 → 跳回列表 |
| 任一校验失败 | 流程中止，按钮 loading 重置，可重试 |
| 抹零弹窗取消 → 再次保存 | costNewList 不含抹零行，不会重复计入 |
| 低风险运单 | 跳过风控弹窗直接提交 |
| 中高风险 → 确认 → allowPass='100' | 提交成功 |
| 中高风险 → 确认 → allowPass≠'100' | 展示失败原因 |
| 接口超时/断网 | btnLoading 重置（finally 保证），可重试 |

**边界情况**：
- 用户快速双击提交按钮 → `btnLoading = true` 在 finally 前不会被重置，天然防重
- 保存流程不走风控 → `_runRiskAndSubmit` 内 `action === 'save'` 时只调 postData 不调风控
- 亏吨弹窗（`cargoCompensationConfirmFlag='1'`） → 在 `_validateCargoLoss` 中处理，返回 false 中止

**收益**：
- 新增/删除校验步骤只改 `_executeFlow` 里一行，不需要通读 200 行链路找插入点
- `btnLoading` 统一在 finally 重置，不再有"某个分支忘了重置"的风险
- 修复 3 个线上隐性 bug

## 第四波：通信方式统一（代码规范）

### T6 — 风控弹窗 provide/inject 改为 emit

**文件**：`edit.vue` + `RiskWarningDialog.vue`

**现状**：`edit.vue` 通过 `provide()` 将 3 个方法（`updateRiskWarning`、`handleCloseRiskDialog`、`handleSubmitRiskDialog`）注入给 `RiskWarningDialog`，子组件通过 `inject` 反向调用父组件方法。数据流方向是向上的，追踪触发时机需要跳到子组件里找 inject 调用点。`handleSubmitRiskDialog` 里还有一个 `riskResult = null` 的无效变量，导致失败提示永远走通用分支。

**目标**：`RiskWarningDialog` 通过 `$emit('submit')` / `$emit('close')` / `$emit('update')` 向外通信，`edit.vue` 在模板上用 `@submit` / `@close` / `@update` 监听。数据流符合 Vue 单向原则，追踪事件路径只需看模板上的 `@` 绑定。删除无效的 `riskResult` 变量。

**现象（开发痛点）**：追踪风控弹窗的"确认"行为时，`RiskWarningDialog` 里通过 inject 反向调用父组件方法，要跳到子组件里找 inject 调用点才能确定触发时机。违反 Vue 单向数据流，新人看代码容易困惑。

**改法**：

RiskWarningDialog 内部：
```js
// 改前：this.handleSubmitRiskDialog()   （inject 调用父方法）
// 改后：this.$emit('submit')
```

edit.vue 模板：
```html
<RiskWarningDialog
  :visible.sync="riskWarningDialogVisible"
  :waybill-list="highMidWaybillList"
  @close="handleCloseRiskDialog"
  @submit="handleSubmitRiskDialog"
  @update="updateRiskWarning"
/>
```

edit.vue 删除 `provide()` 整个声明。

**前置确认**：改动前搜索 `RiskWarningDialog` 在 `index.vue` 或其他文件中是否也通过 provide/inject 调用。如有，需同步改造。

**测试重点**：
| 场景 | 预期 |
|------|------|
| 中高风险运单提交 | 风控弹窗正常弹出 |
| 点击关闭 | 弹窗关闭，不触发提交 |
| 点击确认且 allowPass='100' | 关闭弹窗，继续提交 |
| 点击确认且 allowPass≠'100' | 展示失败提示 |
| 单条处理完后无中高风险 | 自动关闭弹窗并提交 |
| index.vue 使用场景（如有） | 不受影响 |

**边界情况**：
- `handleSubmitRiskDialog` 里 `riskResult = null` 的无效变量 → 在 T5 中已修复，此处确认删除干净
- `warnTable` 弹窗仍用 `$refs` 调用 → 本次不改（风格不同但不阻塞），后续有需求再统一

**收益**：数据流清晰，追踪事件路径从"跨文件找 inject"变为"模板上看 @submit"。符合 Vue 最佳实践。

## 第五波：消除重复代码（长期维护成本）

### T7 — 共用逻辑提取 mixin（useWaybillDetail.js）

**文件**：新建 `useWaybillDetail.js` + 改动 `edit.vue` + `read.vue`

**现状**：`edit.vue`（1686 行）和 `read.vue`（944 行）有约 430 行完全相同的代码（`getDetails`、`handleLookImg`、`getRecordPaging`、`loadStatusOption`、`handleViewCarInfo` 5 个方法 + 运单信息表格模板）。任何接口字段变更必须同步两个文件，漏改一边就是 bug。`handleLookImg` 内部是 3 层嵌套 if/else 做 transportType → wabillStatus 映射，可读性差。`once` 标志位在 `$nextTick` 里重置，不能保证所有异步回填完成。

**目标**：公共逻辑集中在 `useWaybillDetail.js` 一个文件里维护，`edit.vue` 和 `read.vue` 各自只保留差异部分（edit 保留保存/提交/风控/亏吨，read 保留只读展示和返回）。`handleLookImg` 的映射改为二维常量表，一眼可查。`once` 在所有异步步骤完成后才释放，杜绝联动提前触发。

**现象（开发痛点）**：`edit.vue` 和 `read.vue` 有约 430 行完全相同的代码。任何接口字段变更都要同步改两个文件，忘改一边就是 bug。

**重复清单**：

| 方法 | 行数 | 说明 |
|------|------|------|
| `getDetails` | ~120 行 | 拉取详情 + 数据拍平 |
| `handleLookImg` | ~50 行 | 附件查看，含 3 层嵌套 if/else |
| `getRecordPaging` | ~30 行 | 操作记录分页 |
| `loadStatusOption` | ~30 行 | 枚举字典加载 |
| `handleViewCarInfo` | ~20 行 | 跳转车辆监控 |
| 运单信息表格模板 | ~150 行 | `bmsWaybillPoolList` 渲染 |

**改法**：

1. 新建 `useWaybillDetail.js`（mixin），迁入上述 5 个方法
2. `handleLookImg` 里 3 层 if/else 改为二维映射表：
```js
const WAYBILL_STATUS_MAP = {
  '3': { '1': '410', '2': '420', '3': '500' },
  '6': { '1': '410', '2': '420', '3': '500' },
  '1': { '1': '410', '2': '430', '3': '430' },
  '2': { '1': '410', '2': '430', '3': '430' },
  default: { '1': '400', '2': '420', '3': '500' }
}
```
3. `getDetails` 里 `once = false` 改为在所有异步步骤完成后执行（修复 `$nextTick` 不保证所有异步回填完成的问题）
4. `edit.vue` 和 `read.vue` 各自 `mixins: [useWaybillDetail]`，只保留差异逻辑

**同时修复的 bug**：`once` 标志位在 `$nextTick` 里重置，不能保证所有异步回填完成，watch 可能提前触发运费联动覆盖正确值。

**测试重点**：
| 场景 | 预期 |
|------|------|
| edit.vue 进入 | 详情正常回填（结算主体/银行卡/运单信息表格） |
| read.vue 进入 | 所有字段只读展示正确 |
| 各 transportType 的附件查看 | 弹出预览窗口，图片正确 |
| 操作记录翻页 | 分页数据正确 |

### T8 — 运单信息表格组件化（WaybillInfoTable.vue）

**文件**：新建 `components/WaybillInfoTable.vue` + 改动 `edit.vue` + `read.vue`

**依赖**：T7 完成后执行（数据方法已在 mixin 里）

**现状**：`edit.vue` 和 `read.vue` 模板里各有约 150 行完全相同的运单信息 `el-table`（运单号跳转、附件查看、车辆监控等列），两边独立维护。表格列需要调整时（比如新增一列、改列宽、改显隐条件），必须同步改两个文件的模板，且无法通过 diff 验证两边是否一致。

**目标**：运单信息表格抽为独立组件 `WaybillInfoTable.vue`，`edit.vue` 和 `read.vue` 各用一行 `<WaybillInfoTable ... />` 引入。表格列的调整只改组件一处，两个页面自动同步。edit.vue 从约 1400 行降至约 900 行，read.vue 从约 700 行降至约 400 行。

**现象**：`edit.vue` 和 `read.vue` 模板里约 150 行完全相同的运单信息 `el-table`，包含运单号跳转、附件查看、车辆监控等交互。

**改法**：

组件接口设计：
```
props:
  list           Array    bmsWaybillPoolList
  detail         Object   运单详情（状态判断用）
  controlBtn     Object   按钮权限控制
  dictionary     Object   枚举字典
emits:
  look-waybill   row      运单号点击跳转
  look-img       scope, btnType   附件查看
  view-car       row      车辆监控
```

`edit.vue` / `read.vue` 模板中约 150 行表格代码替换为：
```html
<WaybillInfoTable
  :list="bmsWaybillPoolList"
  :detail="detail"
  :control-btn="controlBtn"
  :dictionary="dictionary"
  @look-waybill="handleLookWaybill"
  @look-img="handleLookImg"
  @view-car="handleViewCarInfo"
/>
```

**测试重点**：
| 场景 | 预期 |
|------|------|
| 运单信息表格渲染 | 各列数据正确（运单号、车牌、吨位、状态） |
| 运单号点击 | 跳转运单详情页 |

| controlBtn 无权限 | 对应按钮不展示 |

**边界情况**：
- 表格列的 `v-if` 条件依赖 `detail` 里的字段 → 确保 prop 传入时 detail 已有值，或组件内做空值兜底
- `bmsWaybillPoolList` 为空 → 表格展示空状态

**收益**：
- edit.vue 从约 1400 行降至约 900 行，read.vue 从约 700 行降至约 400 行
- 表格列调整只改一个组件文件
