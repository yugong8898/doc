# 运单结算处理模块 — 优化方案

> 目标：方便后续功能迭代和维护，局部渐进式推进，不做一刀切的大重构。
>
> 优先级说明：**P0** 随时可做 / **P1** 触碰该文件时顺手做 / **P2** 需单独排期+测试

---

### P2 收益（专项排期）

| # | 收益类型 | 具体说明 |

## 一、问题全景（按维度分析）

### 1.1 可读性问题

| 位置 | 问题描述 |
|------|---------|
| `read.vue` 运单号列 | `:class="[controlBtn.btnLookDetail ? 'blue pointer' : 'blue pointer']"` 两侧结果相同，死条件 |
| `edit.vue` 提交链路 | `handleSave → changeWaybillStatus → beforeChangeWaybill → checkOrder → riskManagement → postSubmitAfterRiskPass` 5级深调用链，每级有副作用，追 bug 要跳5个方法 |
| `edit.vue` | import 里有被注释掉的导入（`ImgPreview`、`LoseAndRiseViewText`、`handleCheckQuantity`、`getNewFreightAmount`），无说明 |
| `edit.vue` `getDetails` | 方法体超 100 行，包含数据拍平、状态判断、风控触发、绑卡检查多件事 |
| `CostMessageDetails.vue` `handleGetSummaries` | 小计行的 `confirmRemark` 列展示逻辑里嵌套了分次付款提示和货物保障提示，两段逻辑交织在一处 |
| `CostMessageDetails.vue` | `mounted` 和 `detailObj` watch 里有两段几乎完全相同的 `newList` 初始化代码，差别只在于全流程处理 |

### 1.2 魔法数字问题

| 位置 | 裸数字/字符串 | 含义 |
|------|------------|------|
| `edit.vue` / `read.vue` `busStatus` 判断 | `'300'/'310'/'320'/'330'/'340'` | 待处理/已保存/结算通过/待预审/预审不通过 |
| `edit.vue` `settlementFlag` 判断 | `'0'/'1'/'2'` | 正常/转移/不结算 |
| `CostMessageDetails.vue` | `[101, 102]` orderNumber | 应收/应付抹零行 |
| `CostMessageDetails.vue` | `'200'/'600'` costType | 应付/其他应付 |
| `CostMessageDetails.vue` | `'F10101'/'F10102'` costCode | 固定扣款（按金额/按比例） |
| `edit.vue` `riskManagement` | `'200'/'300'/'400'` | 低/中/高风险等级 |
| `index.vue` `handleSelectable` | `['120','210','300','310','340']` | 可操作状态列表 |

### 1.3 结构问题

| 位置 | 问题描述 |
|------|---------|
| `edit.vue` 与 `read.vue` | 两个文件有大量完全相同的代码：`getDetails`、`handleLookImg`、`getRecord`/`getRecordPaging`、`loadStatusOption`、`handleViewCarInfo`、运单信息表格模板（约400行）。接口字段变化时两边都要改 |
| `edit.vue` `provide` | 将 `updateRiskWarning`、`handleCloseRiskDialog`、`handleSubmitRiskDialog` 通过 `provide` 注入到 `RiskWarningDialog`，子组件反向调用父组件方法，数据流方向不清晰 |
| `components/add.conf.js` | 整个文件只定义了一个"返回"按钮配置，体量极小但单独占一个 mixin，增加无谓的文件跳转 |
| `edit.vue` `data` | `wayStatus` 对象（`{ 1: '公路', 4: '铁路', 5: '水路', ... }`）和 `statusMap` 对象内联在 data 里，应提取为常量 |

### 1.4 数据流问题

| 位置 | 问题描述 |
|------|---------|
| `CostMessageDetails.vue` `newList` | 有两处初始化：`mounted` 一次，`detailObj` watch 一次。`mounted` 的结果是"过渡态"，几乎立刻被 watch 覆盖。两处逻辑存在细微差异（全流程时 `mounted` 不强制 premiumAmount 为 0，watch 会强制），容易踩坑 |
| `CostMessageDetails.vue` `newList[0]` | `countMoney` 方法硬编码写 `this.newList[0]`，但当 `companyType === 1` 时 `newList[0]` 是货物保障服务行而不是运费差价行，存在写错位置的风险 |
| `edit.vue` `once` 标志位 | `once = true` 时禁止 watch 触发运费重算，`once = false` 时开放。设置时机分散在 `getDetails`、`updateOnce`、`mounted` 多处，容易搞混 |
| `edit.vue` `getDetails` | 方法内直接修改 `this.formModel`、`this.detail`、`this.bmsWaybillPoolList` 多个顶层响应式数据，副作用较多，测试和追踪困难 |

### 1.5 健壮性问题

| 位置 | 问题描述 |
|------|---------|
| `CostMessageDetails.vue` `countMoney` | `newList[0]` 硬编码下标，`companyType === 1` 时会写错行（见上） |
| `edit.vue` 提交前校验 | 抹零逻辑（`diffAmount !== 0` 时追加行）在校验链路的中间位置，若用户取消抹零弹窗，后续校验已经部分执行，状态不一致 |
| `edit.vue` `getRiskDetails` | 风控查询结果直接写 `this.highMidWaybillListPage`，但旧数据不清空，刷新时可能残留上次的风控结果 |
| `edit.vue` `getDetails` | 有 `if (this.loading) return` 防重，但 `loading` 在 catch 分支里没有重置为 false（只在 `if (datas)` 里重置），接口报错后页面会永久 loading |

**`_executeSettlementFlow` 目标结构**：
```js
async _executeSettlementFlow(action) {
  // 1. 费用明细校验
  if (!await this.$refs.CostMessageDetails.commit()) return
  // 2. 油气费不能超运费
  if (!this._validateOilGasCost()) return
  // 3. 抹零处理（差值不为0时弹窗，取消则中止）
  if (!await this._handleZeroMode()) return
  // 4. 表单字段校验
  if (!await this._validateForm()) return
  // 5. 运单状态校验
  if (!await this._checkOrderStatus()) return
  // 6. 风控测算 + 提交
  await this._runRiskAndSubmit(action)
}
```

**`newList` 初始化唯一入口**：
```js
// 改前：mounted + watch 两处
// 改后：只在 detailObj watch 里
watch: {
  detailObj(val) {
    this.$nextTick(() => {
      if (this.newList.length === 0 || this.type !== 'edit') {
        this._buildNewList(val)       // 首次 / 查看模式：完整重建
      } else {
        this._updateOilGasOnly(val)   // 编辑模式已初始化：只更新油气行
      }
    })
  }
}
```
