# 运单结算处理模块 — 优化方案

> 目标：方便后续功能迭代和维护，局部渐进式推进，不做一刀切的大重构。

---

## 总体原则

- **P0（随时可做）**：改动不影响业务逻辑，可在日常迭代中顺手处理，30min 内能完成
- **P1（触碰时做）**：下次需求触碰到该代码时顺手重构，不单独排期
- **P2（专项排期）**：涉及组件拆分/通信调整，需单独排期 + 充分测试

## P1 — 提交流程显式化

### 任务 5：将 edit.vue 的提交链路改为线性 async/await

**文件**：`src/views/userBase/waybillSettlement/edit.vue`

**问题**：当前提交流程是一条 5 级深的调用链，每级还有副作用（弹窗/抹零/风控），追踪一个 bug 要跳 5 个方法：

```
handleSave/handleSubmit
  → changeWaybillStatus(status, type)
      → beforeChangeWaybill()
          → checkOrder()
              → riskManagement()
                  → postSubmitAfterRiskPass()
```

新增一个前置校验（如"结算重量不能为0"）需要找到正确的插入位置，极易改错。

**触发时机**：下次提交流程有新需求（新增校验步骤、调整抹零逻辑）时执行。

**目标结构**：

```js
// 统一入口，流程一目了然
async handleSave() {
  await this._executeSettlementFlow('save')
},
async handleSubmit() {
  await this._executeSettlementFlow('submit')
},

async _executeSettlementFlow(action) {
  // Step 1：费用明细校验
  const feeValid = await this.$refs.CostMessageDetails.commit()
  if (!feeValid) return

  // Step 2：油气费校验（司机运费 ≥ 实付油气费）
  if (!this._validateOilGasCost()) return

  // Step 3：付款等式校验（已迁移至 SettlementInfo 内部，此处跳过）

  // Step 4：抹零处理（差值不为0时弹窗确认）
  const zeroConfirmed = await this._handleZeroMode()
  if (!zeroConfirmed) return

  // Step 5：表单校验
  const formValid = await this._validateForm()
  if (!formValid) return

  // Step 6：运单状态校验
  const orderValid = await this._checkOrderStatus()
  if (!orderValid) return

  // Step 7：风控测算 + 实际提交
  await this._runRiskAndSubmit(action)
},

// 各步骤方法（职责单一，独立可测）
_validateOilGasCost() { /* ... */ },
async _handleZeroMode() { /* ... */ },
async _validateForm() { /* ... */ },
async _checkOrderStatus() { /* ... */ },
async _runRiskAndSubmit(action) { /* ... */ }
```

**好处**：
- 新增校验步骤只需在流程里加一行 `if (!await this._validateXxx()) return`
- 调试时可以直接在 `_executeSettlementFlow` 打断点，不需要跳多层

**验证清单**：

```
□ 保存（310）：费用明细校验通过后保存，跳回列表
□ 提交（320）：全部校验通过 + 风控通过后提交，跳回列表
□ 抹零确认弹窗：差值 > 0 / < 0 时弹窗，取消后流程中止
□ 油气费超运费时：给出提示，流程中止
□ 运单状态异常时：warnTable 弹窗展示，流程中止
□ 中高风险时：RiskWarningDialog 弹窗，确认后继续提交
□ 风控弹窗取消：流程中止，已保存的数据保留
```

## P2 — CostMessageDetails newList 初始化时机统一

### 任务 7：消除 CostMessageDetails.vue 的双重初始化

**文件**：`src/views/userBase/waybillSettlement/components/CostMessageDetails.vue`

**问题**：`newList`（运费差价/货物保障/油气优惠汇总区数据）有两处初始化逻辑：
1. `mounted()`：首次构建，**未处理全流程**
2. `detailObj` watch 的"完整重建分支"：**会处理全流程**，用 0 覆盖 `premiumAmount`

两处逻辑不一致，`mounted` 的结果是"过渡态"，马上被 watch 覆盖。虽然最终显示正确，但逻辑上存在隐患，且非常难以理解。

**触发时机**：下次 newList 相关逻辑有变动时执行。

**目标结构**：删除 `mounted` 中的 newList 初始化，只保留 `detailObj` watch 的完整重建分支作为唯一初始化入口。

```js
// 改前：两处初始化
mounted() {
  this.$nextTick(() => {
    this._buildNewList()  // ← 删除这里的调用
  })
},
watch: {
  detailObj(val) {
    this.$nextTick(() => {
      if (this.newList.length === 0 || this.type !== 'edit') {
        this._buildNewList()  // ← 保留唯一入口
      } else {
        this._updateOilGasOnly()
      }
    })
  }
}

// 改后：只在 detailObj 变化时初始化
mounted() {
  // newList 由 detailObj watch 负责初始化，此处不处理
},
```

**验证清单**：

```
□ 进入 edit 页面：运费差价 / 货物保障 / 油气优惠行正常展示
□ 进入 read 页面：同上
□ 全流程运单：货物保障服务金额显示 0
□ 编辑模式下保存再返回：newList 不会被覆盖重置
□ 多次切换运单详情：newList 正确刷新
```

## P2 — 风控逻辑下沉

### 任务 8：将 index.vue 的风控回调从 provide/inject 改为更明确的事件总线或 ref 调用

**文件**：`src/views/userBase/waybillSettlement/index.vue` 和 `edit.vue`

**问题**：风控弹窗（`RiskWarningDialog`）通过 `provide` 注入 `handleCloseRiskDialog`、`handleSubmitRiskDialog`、`updateRiskWarning` 等回调，但这三个回调的实际业务逻辑（提交运单/更新页面状态）都在父组件里，子组件通过 inject 反向调用父组件方法，导致数据流方向不清晰，调试时需要在父子两层之间反复跳转。

**触发时机**：下次风控逻辑有改动时执行。

**目标**：`RiskWarningDialog` 改为通过 `$emit` 发出事件，父组件监听处理：

```html
<!-- 父组件模板 -->
<RiskWarningDialog
  :visible.sync="riskDialogVisible"
  :waybill-list="highMidWaybillList"
  @close="handleCloseRiskDialog"
  @submit="handleSubmitRiskDialog"
  @update="updateRiskWarning"
/>
```

**验证清单**：

```
□ 批量提交有中高风险运单时，风控弹窗正常弹出
□ 点击关闭：弹窗关闭，summaryFinalSettleResult 正常汇总
□ 点击确认：可通过的运单正常提交，不可通过的记入失败
□ updateRiskWarning 回调后，页面风控展示信息同步更新
□ edit.vue 的风控逻辑不受影响
```
