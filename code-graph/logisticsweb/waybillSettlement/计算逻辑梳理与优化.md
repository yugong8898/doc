# 运单结算 — 计算逻辑全景梳理与优化方案

> 本文档覆盖 `edit.vue` + `CostMessageDetails.vue` + `endOfCalculation.js` 中所有涉及金额计算的逻辑。

---

## 一、计算逻辑全景图

```
用户操作触发
├── 修改单价/结算数量 → 运费联动计算
├── 修改确认金额 → 小计联动 → 运费差价联动 → 油气优惠联动 → 抹零联动
├── 修改装卸重量 → 重新拉详情 → 全部重算
└── 保存/提交 → 抹零计算 → 提交前运费差价修正

核心计算链：
  单价 × 结算数量 = 运费（transportCost）
       ↓
  费用明细确认金额汇总 = 小计（subtotal）
       ↓
  小计 ÷ (1 - 差价率/100) = 合计（total）
       ↓
  合计 - 小计 - 货物保障 = 运费差价（serviceCharge）
       ↓
  应付合计经抹零处理 = 抹零后金额（processedTotalAmount）
       ↓
  抹零差额 → 追加应收/应付抹零行
       ↓
  油气优惠 = 接口实时计算（参数为小计金额）
```

---

## 二、各计算点详细说明

### 2.1 运费计算（edit.vue watch）

**触发时机**：`formModel.unitRice` 或 `formModel.settlementQuantity` 变化时

**公式**：
```
运费 = fomatFloatRoundMode(单价 × 结算数量, 2, roundingMode)
```

**保护机制**：`once` 标志位为 true 时跳过计算，防止数据回填时触发联动

**涉及代码**：
- `edit.vue` L521~L534（两个 watch）
- `edit.vue` L1100 `updBusCostInput` 方法（手动触发同样的计算）

**问题点**：
1. `once` 标志位设置时机不稳定：在 `getDetails` 里设 `true`，在 `updateOnce` 里设 `false`（`$nextTick`），但 `getDetails` 内有多个异步步骤尚未完成
2. `updateOnce` 对 POW 开头 + supplyType=300 的运单不解锁（永远 once=true），意味着这类运单不走自动运费计算
3. watch 设了 `immediate: true`，`mounted` 时 `once` 还没被设置，首次可能误触发

---

### 2.2 小计计算（CostMessageDetails.vue handleGetSummaries）

**触发时机**：el-table 渲染/数据变化时自动调用（summary-method）

**公式**：
```
小计 = Σ(应付费用确认金额) - Σ(应收费用确认金额)
     = Σ(costType∈['200','600'] 的 updBusCost) - Σ(costType∈['100','500'] 的 updBusCost)
```

**排除项**：抹零行（orderNumber 101/102）不在费用明细表格中（它们是保存时动态追加的）

**涉及代码**：CostMessageDetails.vue `handleGetSummaries` 方法

**问题点**：
1. `handleGetSummaries` 是 el-table 的 summary-method，由 Vue 渲染驱动调用，**触发时机不可控**
2. 该方法内同时做了 UI 渲染（返回 sums 数组）和业务计算（给 `this.subtotal` 赋值），职责混合
3. 方法内还调了 `syncFixedDeductionAmounts()`（同步固定扣款），每次表格渲染都会执行

---

### 2.3 运费差价计算（CostMessageDetails.vue countMoney）

**触发时机**：`subtotal` watch 变化 + 编辑模式 + `detailObj.taxRate` 存在时

**公式**：
```
合计 = count(小计, 差价率) + 货物保障服务金额
     = fomatFloatRoundMode(小计 / getBit(1 - 差价率/100, 7), 2, roundingMode) + premiumAmount

运费差价 = 合计 - 小计 - 货物保障服务金额
```

**其中 `getBit(value, 7)` 的作用**：截取小数到 7 位，避免浮点精度问题

**涉及代码**：
- CostMessageDetails.vue `countMoney(cost, rate)` 方法
- CostMessageDetails.vue `count(Amount, Rate)` 方法
- CostMessageDetails.vue `formatService(num, n, type)` 方法（取整）

**问题点**：
1. `this.newList[0]` 硬编码下标 — B1 企业时第 0 项是货物保障服务，不是运费差价
2. `countMoney` 在两处被调用：subtotal watch + `changeWaybillStatus`（保存前修正），触发来源不同但未做区分
3. `count` 和 `formatService` 在 CostMessageDetails 和 endOfCalculation mixin 里**各有一份**，逻辑完全相同但各自维护

---

### 2.4 油气优惠计算（CostMessageDetails.vue updateOilGasDiscountIfNeeded）

**触发时机**：
- 确认金额失焦时（`updBusCostBlur`）
- 抹零计算完成后（`dealZeroModeFn` 末尾）

**公式**：调用后端接口 `calculateOilRebate`，参数为当前小计金额

**接口参数**：
```js
{
  waybillNo: 运单号,
  adjustedAmount: this.subtotal,  // 小计金额
  needDetail: 'true',
  customerRate: this.detailObj.taxRate  // 差价率
}
```

**返回值**：`rebateAmount`（返利金额）→ 更新 newList 中"油气优惠"行的 `confirmMoney`

**问题点**：
1. 每次任何费用确认金额失焦都会调一次接口，高频场景（连续修改多行）会发起大量请求
2. 接口返回兼容逻辑复杂（兼容 `{succeed, model}` 和直接返回两种结构 + catch 里的特殊处理）
3. 全流程运单强制显示 0，但仍然发起接口调用

---

### 2.5 抹零计算（CostMessageDetails.vue dealZeroModeFn）

**触发时机**：
- 确认金额 input 事件（`updBusCostInput` → debounce 500ms）
- `getZeroByBusId` 获取抹零方式后
- `updBusCostInput`（edit.vue 结算信息变化时）

**公式**：
```
应付合计 = Σ(应付) - Σ(应收)  // 排除已有的抹零行
抹零后金额 = processNumber(应付合计, zeroByBusIdMode)
差额 = 应付合计 - 抹零后金额

差额 < 0 → 追加 orderNumber=102 的应付行（金额=|差额|）
差额 > 0 → 追加 orderNumber=101 的应收行（金额=|差额|）
差额 = 0 → 不追加
```

**`processNumber` 抹零模式**：
- `'0'`：不抹零
- `'10'`：小数抹零（去掉小数部分）
- `'20'`：个位抹零
- `'30'`：四舍五入到个位，个位 <5 取 0，≥5 取 5

**涉及代码**：
- CostMessageDetails.vue `dealZeroModeFn()` — 实时预览抹零行
- edit.vue `changeWaybillStatus()` — 保存前正式计算并追加

**问题点**：
1. **两处独立计算，逻辑重复**：`dealZeroModeFn`（实时预览）和 `changeWaybillStatus`（保存前）各算一次
2. `dealZeroModeFn` 直接 push 到 `params.model`（费用明细数组），保存时 `changeWaybillStatus` 又往 `costNewList` 里 push 一份，可能重复
3. 保存前的 `changeWaybillStatus` 取到的 `payableTotalArr` 已经 filter 掉了 101/102，但如果 `dealZeroModeFn` 之前追加的行没被清理干净，合计可能偏差
4. 抹零弹窗取消后，`costNewList` 里已 push 的抹零行没有清理

---

### 2.6 保存前运费差价修正（edit.vue changeWaybillStatus）

**触发时机**：点击保存/提交按钮后，抹零计算完成时

**逻辑**：
```js
if (this.detail.taxRate != null && this.zeroByBusIdMode && diffAmount !== 0) {
  const currentTaxRate = this.$refs.CostMessageDetails.newList[0]?.confirmRemark || this.detail.taxRate
  this.$refs.CostMessageDetails.countMoney(processedTotalAmount, currentTaxRate)
}
```

**含义**：抹零后金额变了，运费差价也需要基于新金额重新算一遍

**问题点**：
1. 又是 `newList[0]` 硬编码下标
2. `processedTotalAmount` 作为 `cost` 参数传入 `countMoney`，但此时小计还是原来的值，`subtotal` watch 可能被意外触发再算一遍

---

### 2.7 固定扣款同步（CostMessageDetails.vue syncFixedDeductionAmounts）

**触发时机**：
- 运费（transportCost）变化时
- 确认金额输入时（updBusCostInput）
- 小计计算时（handleGetSummaries 内部调用）

**公式**：
```
F10102（按比例）：运费确认金额 × 系数（item1）
F10101（按固定金额）：直接用 busCost
```

**涉及代码**：CostMessageDetails.vue `syncFixedDeductionAmounts()`

**问题点**：
1. 被 `handleGetSummaries` 调用，而 `handleGetSummaries` 是 summary-method，执行频率不可控
2. 运费行确认金额变化 → 固定扣款金额变 → 又触发小计变化 → 可能形成循环

---

### 2.8 全流程运费计算（CostMessageDetails.vue doCalculateFullProcessFreight）

**触发时机**：`countMoney` 方法末尾调用（debounce 300ms）

**条件**：编辑模式 + `fullProcessFlag === 10`

**逻辑**：调用后端接口 `doCalculateFullProcessFreight`，参数为当前小计

**返回**：`fpFreightPriceRate`（差价率）、`payableFpFreight`（客户应付运费）、`fpFreightDiffAmt`（运费差价金额）

**问题点**：
1. 接口返回的差价相关字段通过 `$emit('updateDetail')` 传回 edit.vue，再更新 `this.detail`
2. detail 变化又触发 detailObj watch，但走的是"已初始化保护分支"不重建 newList — 逻辑正确但不直观

## 四、计算触发关系（连锁反应图）

```
用户修改单价/结算数量
  → [watch] transportCost = 单价 × 数量（roundingMode）
    → [watch transportCost] 费用明细中 orderNumber<11 的行 updBusCost 更新
      → [el-table rerender] handleGetSummaries 重算小计
        → [watch subtotal] countMoney(subtotal, taxRate) 重算运费差价
          → doCalculateFullProcessFreight（全流程时）
      → syncFixedDeductionAmounts 固定扣款同步
        → 可能再次触发 handleGetSummaries

用户修改费用明细确认金额
  → [input] updBusCostInput → debounce dealZeroModeFn
  → [blur] updBusCostBlur → updateOilGasDiscountIfNeeded（油气优惠接口）
  → [el-table rerender] handleGetSummaries 重算小计
    → 同上链路

保存/提交
  → changeWaybillStatus
    → 费用校验 → 油气费校验
    → changeMoney（格式化）
    → 计算 payableTotalAmount（应付合计）
    → processNumber 抹零
    → countMoney（修正运费差价）
    → 弹窗确认抹零
    → beforeChangeWaybill → checkOrder → riskManagement → postData
```

## 五、核心问题总结

| 问题类型 | 具体表现 | 严重度 |
|---------|---------|--------|
| 职责混合 | `handleGetSummaries` 同时做 UI 渲染和业务计算（赋值 subtotal） | 高 |
| 重复计算 | `count`/`formatService`/`getBit` 在 mixin 和组件里各一份 | 中 |
| 双源抹零 | `dealZeroModeFn`（实时预览）和 `changeWaybillStatus`（保存前）各自独立计算 | 高 |
| 硬编码下标 | `newList[0]` 在多处假设"运费差价"永远是第 0 项 | 高 |
| 隐式联动 | subtotal 变化 → countMoney → emit updateDetail → detail 变 → watch 触发 → 又回到 subtotal | 高 |
| 触发时机不可控 | summary-method 由 Vue 渲染驱动，无法精确控制何时执行 | 中 |
| 高频接口 | 每次 blur 都调 `calculateOilRebate`，连续修改多行时大量请求 | 中 |
| once 不稳定 | `$nextTick` 不保证所有异步完成，watch 可能提前触发 | 高 |

## 六、优化方案

### 方案 A：小计计算与 UI 渲染分离

**问题**：`handleGetSummaries` 既是 UI 渲染函数又是业务计算入口，职责不清且触发时机不可控。

**方案**：
1. 新增独立的 `calcSubtotal()` 方法，纯粹计算小计
2. `handleGetSummaries` 只负责格式化展示，读取已算好的 `this.subtotal`
3. 在需要的地方（`updBusCostInput`、`updBusCostBlur`、`transportCost` watch）显式调用 `calcSubtotal()`

**目标代码**：
```js
calcSubtotal() {
  const sum = this.params.model.reduce((total, item) => {
    if ([101, 102].includes(item.orderNumber)) return total
    if (['200', '600'].includes(`${item.costType}`)) {
      return total.plus(+item.updBusCost || 0)
    } else {
      return total.minus(+item.updBusCost || 0)
    }
  }, new BigNumber(0))
  this.subtotal = Number(sum)
}
```

**收益**：
- 计算时机完全可控，不再依赖 el-table 渲染
- 调试时能精确断点
- `handleGetSummaries` 变为纯展示函数，无副作用

**影响范围**：仅 CostMessageDetails.vue
**风险**：低

---

### 方案 B：运费差价计算安全化

**问题**：`countMoney` 硬编码 `newList[0]`，B1 写错行；且在两处被不同参数调用。

**方案**：
1. 按 `costName` 查找目标行，找不到时安全退出
2. 将保存前修正和 subtotal watch 的调用统一为同一逻辑

**目标代码**：
```js
countMoney(cost, rate) {
  if (!rate) rate = 0
  const total = this.count(cost, rate) + (this.detailObj.premiumAmount || 0)
  const serviceRateMoney = new BigNumber(total - cost - (this.detailObj.premiumAmount || 0)).toFixed(2)
  
  const freightDiffRow = this.newList.find(i => i.costName === '运费差价')
  if (freightDiffRow) {
    this.$set(freightDiffRow, 'confirmMoney', serviceRateMoney)
    this.$set(freightDiffRow, 'confirmRemark', rate)
  }
  
  this.$emit('newServiceCharge', serviceRateMoney)
  this.doCalculateFullProcessFreight()
}
```

**保存前修正也改为按名查找**：
```js
// edit.vue changeWaybillStatus 中
const freightDiffRow = this.$refs.CostMessageDetails.newList.find(i => i.costName === '运费差价')
const currentTaxRate = freightDiffRow?.confirmRemark || this.detail.taxRate
this.$refs.CostMessageDetails.countMoney(processedTotalAmount, currentTaxRate)
```

**收益**：
- 彻底消除 B1 企业写错行的 bug
- 代码意图明确

**影响范围**：CostMessageDetails.vue + edit.vue（1 行）
**风险**：低

---

### 方案 C：抹零计算单一来源

**问题**：`dealZeroModeFn`（实时预览）和 `changeWaybillStatus`（保存前）各自独立计算，逻辑重复且可能不一致。

**方案**：
1. `dealZeroModeFn` 仅负责**实时预览**：在费用明细表格中展示/更新抹零行
2. `changeWaybillStatus` 保存前**不再独立计算**，直接使用费用明细表格中已有的抹零行数据
3. 取消保存时 push 到 `costNewList` 的逻辑，改为直接从 `params.model` 中收集（包含抹零行）

**目标流程**：
```
编辑过程中：
  修改确认金额 → updBusCostInput → dealZeroModeFn → 表格中实时展示抹零行

保存时：
  changeWaybillStatus → 从 params.model 取所有行（含抹零行） → 校验 → 保存
  不再重复计算，不再 push
```

**关键改动**：
```js
// edit.vue changeWaybillStatus 改后
const newlist = this.changeMoney(CostDetails)
this.costNewList = newlist  // 直接用，包含 dealZeroModeFn 已追加的抹零行

// 删除以下逻辑：
// const payableTotalAmount = ...
// const processedTotalAmount = ...
// this.costNewList.push(...)
// this.dialogConfirmZeroInfo(...)

// 改为：检查是否有抹零行，有则弹窗确认
const zeroRow = this.costNewList.find(i => [101, 102].includes(i.orderNumber))
if (zeroRow && this.zeroByBusIdMode) {
  const confirmed = await this.confirmZeroDialog(zeroRow)
  if (!confirmed) return
}
this.beforeChangeWaybill()
```

**收益**：
- 抹零计算只有一个来源（`dealZeroModeFn`），消除两处不一致
- 保存时不再重复计算，逻辑简化约 30 行
- 抹零取消后不需要清理 `costNewList`（因为没有额外 push）

**影响范围**：edit.vue changeWaybillStatus（改动较大）
**风险**：中高 — 需完整测试保存/提交各路径

**边界场景**：
- 费用明细为空时保存
- 抹零方式为"不抹零"时
- 差额为 0 时（不应有抹零行）
- 连续修改后 dealZeroModeFn 的 debounce 还没执行就点了保存

### 方案 D：油气优惠接口防抖优化

**问题**：每次确认金额 blur 都调接口，连续修改多行时大量无效请求。

**方案**：
1. `updateOilGasDiscountIfNeeded` 加 debounce（500ms），连续操作只发一次
2. 全流程运单（`fullProcessFlag === 10`）跳过接口调用，直接展示 0
3. 加 loading 态（防止旧请求覆盖新请求结果）

**目标代码**：
```js
updateOilGasDiscountIfNeeded: debounce(async function() {
  // 全流程强制为 0，不调接口
  if (this.detailObj.fullProcessFlag === 10) return
  
  const oilAmount = this.detailObj?.oilAmount || this.detailObj?.bmsWaybillPoolExt?.oilAmount
  if (!oilAmount || Number(oilAmount) <= 0) return
  
  // 用请求标记避免旧请求覆盖新请求
  const requestId = ++this._oilGasRequestId
  
  const res = await calculateOilRebate({
    waybillNo: this.detailObj.waybillId,
    adjustedAmount: this.subtotal,
    needDetail: 'true',
    customerRate: this.detailObj.taxRate
  })
  
  // 旧请求结果丢弃
  if (requestId !== this._oilGasRequestId) return
  
  // 更新...
}, 500)
```

**收益**：
- 减少无效接口调用（连续修改 5 行 → 从 5 次降到 1 次）
- 避免竞态条件（旧请求慢于新请求时不会覆盖）

**影响范围**：仅 CostMessageDetails.vue
**风险**：低

### 方案 E：once 标志位稳定化

**问题**：`once` 在 `$nextTick` 里重置为 false，但 `getDetails` 内有多个异步步骤（`checkMemberStatus`、`getRiskDetails` 等），watch 可能提前触发。

**方案**：
1. `once` 的重置改为在 `getDetails` 的所有异步步骤完成之后
2. 不再依赖 `$nextTick`，而是用 `await` 保证顺序

**目标代码**：
```js
async getDetails() {
  if (this.loading) return
  this.loading = true
  this.once = true  // 锁住 watch
  
  try {
    const [err, datas] = await postWaybillDetails(params).then(...)
    if (!datas) return
    
    // ... 所有数据回填 ...
    
    // 异步步骤
    if (this.detail?.freightType === '2') {
      await this.checkMemberStatus(this.detail.settlementId)
    }
    this.getRiskDetails()  // 这个不 await（不影响 watch 联动）
    
    // 所有回填完成后才解锁
    this.$nextTick(() => {
      this.once = this.shouldKeepOnce()  // POW+300 保持锁定
    })
  } finally {
    this.loading = false
  }
}

shouldKeepOnce() {
  return this.detail.orderId?.includes('POW') && this.detail?.supplyType === '300'
}
```

**收益**：
- 运费联动只在数据完全回填后才生效
- POW 特殊逻辑显式化，不再散落在另一个方法里

**影响范围**：edit.vue getDetails + updateOnce
**风险**：中 — 需测试各种运单类型进入编辑页后的行为

---

### 方案 F：消除重复的计算工具方法

**问题**：`count`、`formatService`、`getBit` 在 `CostMessageDetails.vue` 和 `endOfCalculation.js` 里各有一份。

**方案**：
1. CostMessageDetails 里删除 `count`、`formatService`、`getBit` 方法
2. 改为引入 endOfCalculation mixin 或直接 import 工具函数

**两种路径**：
- 路径 1：CostMessageDetails mixins 加入 endOfCalculation → 直接用 `this.count()` 等
- 路径 2：把 `count`、`formatService`、`getBit` 提取为纯函数（不依赖 this），放到 `utils/calculation.js`，两边 import

**推荐路径 2**（纯函数更好测试）：
```js
// utils/calculation.js
import BigNumber from 'bignumber.js'

export function getBit(value, n) {
  let str = value.toFixed(n + 1).toString()
  str = str.split('.')[0] + '.' + str.split('.')[1].substring(0, n)
  return parseFloat(str)
}

export function formatService(num, n, type) {
  const typeObj = { '0': Math.ceil, '1': Math.floor, '4': Math.round }
  const fn = typeObj[type] || Math.ceil
  const newNum = fn(new BigNumber(num).multipliedBy(Math.pow(10, n)).toNumber())
  return new BigNumber(newNum).div(Math.pow(10, n)).toNumber()
}

export function calcTotal(amount, rate, roundingMode) {
  const divisor = getBit(1 - (parseFloat(rate) / 100), 7)
  let num = new BigNumber(amount).div(divisor).toFixed(7)
  return +formatService(num, 2, roundingMode || 0)
}
```

**收益**：
- 两处逻辑合一，改计算规则只改一处
- 纯函数可以写单元测试

**影响范围**：CostMessageDetails.vue + endOfCalculation.js + 新建 utils/calculation.js
**风险**：低
