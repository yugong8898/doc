## 单元 3：getDetails loading bug 修复

### 涉及内容
- `edit.vue` 的 `getDetails` 方法：有 `if (this.loading) return` 防重复调用
- `loading` 在方法开头设为 `true`，但只在 `if (datas)` 分支里重置为 `false`
- **catch 分支里没有重置**，接口报错后 `loading` 永远为 `true`，页面永久 loading

### 调整方向
- 使用 `try/finally` 结构，在 `finally` 里统一 `this.loading = false`
- 删除原来散落在 `if (datas)` 里的 `this.loading = false`

### 影响范围
- 仅 `edit.vue` 的 `getDetails` 方法
- 不改变正常流程行为，只修复异常流程

### 最终结果
```js
async getDetails() {
  if (this.loading) return
  this.loading = true
  try {
    // ... 原有逻辑
  } catch (e) {
    // 错误处理
  } finally {
    this.loading = false
  }
}
```

### 测试回归
- 正常场景：进入编辑页，数据正常加载，loading 消失
- 异常场景：模拟接口 500 错误（Network tab 里 block 请求），页面应展示错误提示且 loading 消失，可重试
- 快速连续点击进入编辑页：不会发起重复请求

### 收益
| 维度 | 说明 |
|------|------|
| 代码数量 | 改动约 5 行 |
| 性能 | 无变化 |
| 可维护 | loading 重置逻辑集中在一处（finally），不再有遗漏风险 |
| 健壮性 | ✅ 修复真实 bug：接口报错后页面不再永久卡死 |

## 单元 5：CostMessageDetails 双重初始化消除

### 涉及内容
- `CostMessageDetails.vue` 的 `mounted` 钩子里初始化 `newList`（运费差价/货物保障/油气优惠三行），约 50 行
- `detailObj` watch 里**再次**初始化 `newList`，逻辑几乎相同但细节不一致
- `mounted` 的结果是"过渡态"，几乎立刻被 watch 覆盖
- 两处差异：全流程运单时 `mounted` 不强制 `premiumAmount=0`，watch 会强制 → 展示不一致

### 调整方向
- 删除 `mounted` 里的 `newList` 初始化逻辑（约 50 行）
- `detailObj` watch 作为 `newList` 唯一初始化入口
- 提取两个明确命名的方法：
  - `_buildNewList(detailObj)`：完整重建（首次进入 / 查看模式）
  - `_updateOilGasOnly(detailObj)`：编辑模式已初始化时，只更新油气行

### 影响范围
- 仅 `CostMessageDetails.vue`
- 不影响父组件 `edit.vue` / `read.vue` 的调用方式

### 最终结果
```js
// mounted 里删除 newList 相关代码，只做非业务初始化
mounted() {
  window.addEventListener('resize', this.handleResize)
  const { type } = this.$route.query
  this.type = type
  this.busBillId = this.params.busBillId
}

// detailObj watch 作为唯一入口
watch: {
  detailObj(val) {
    this.$nextTick(() => {
      if (this.newList.length === 0 || this.type !== 'edit') {
        this._buildNewList(val)
      } else {
        this._updateOilGasOnly(val)
      }
    })
  }
}
```

| 全流程运单 | 货物保障服务金额显示 0（premiumAmount 被强制置 0）|
| companyType=1（B1 企业）| newList 只有货物保障服务行，无运费差价行 |
| 保存后刷新 | newList 正确重建，不残留旧值 |
| 查看模式（read） | 费用明细行展示正确，金额不可编辑 |

### 收益

## 单元 7：保存/提交流程拍平

### 涉及内容
- `edit.vue` 的 5 级调用链：`handleSave → changeWaybillStatus → beforeChangeWaybill → checkOrder → riskManagement → postSubmitAfterRiskPass`
- 每级有副作用，追 bug 要跳 5 个方法
- `.then()` 和 `async/await` 两种异步风格混用
- `btnLoading` 重置时机分散在不同方法里
- 3 个真实 bug：
  1. 抹零弹窗取消后 `costNewList` 里已 push 的抹零行没有清理，下次提交会重复计入
  2. `riskManagement` 开头 `if (this.btnLoading) return` 是死代码（进来前 loading 已被置 false），永远不触发
  3. `handleSubmitRiskDialog` 里 `riskResult = null` 从未赋值，失败原因永远走通用分支

### 调整方向
- 提取 `_executeFlow(action)` 作为统一入口（action = 'save' | 'submit'）
- 每个校验步骤独立为一个 async 方法，返回 boolean
- `action` 直接传入，删除 `btnTypeClicked` 标志位
- 抹零处理封装在 `_handleZeroMode()` 里，取消时内部清理已追加的行再返回 false
- 保留"预保存 → 风控 → 正式提交"的两次接口顺序（风控依赖已固化数据）

### 影响范围
- 仅 `edit.vue` 内部方法重组
- 不改变接口调用参数和顺序
- 不改变组件间通信方式
- 对外行为（保存跳 310、提交跳 320）完全不变

### 最终结果
```js
async _executeFlow(action) {
  this.btnLoading = true
  try {
    if (!await this._validateCostDetails()) return
    if (!this._validateCargoLoss()) return
    if (!this._validateOilGasCost()) return
    if (!await this._handleZeroMode()) return
    if (!await this._validateForm()) return
    if (!await this._checkOrderStatus()) return
    await this._runRiskAndSubmit(action)
  } finally {
    this.btnLoading = false
  }
}
```

各步骤方法：
- `_validateCostDetails()` → 费用明细校验（调用 CostMessageDetails.commit()）
- `_validateCargoLoss()` → 货损未确认时阻止
- `_validateOilGasCost()` → 油气费不能超运费
- `_handleZeroMode()` → 抹零处理（含弹窗确认，取消时清理 costNewList）
- `_validateForm()` → formRef.validate
- `_checkOrderStatus()` → 运单状态接口校验（含 warnTable 弹窗）
- `_runRiskAndSubmit(action)` → 预保存 → 风控测算 → 有风险弹窗 / 无风险正式提交

### 测试回归
| 场景 | 验证点 |

- `edit.vue` 和 `read.vue` 有以下完全重复的方法：
  - `getDetails`（~120 行）：拉取详情 + 数据拍平
