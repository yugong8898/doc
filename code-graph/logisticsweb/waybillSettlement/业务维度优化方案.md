# 运单结算调整（edit.vue）— 业务维度优化方案

> 核心问题：`edit.vue` 1846 行，7 个业务模块的逻辑全部堆在一起，任何一处需求变更都需要通读大量无关代码。
>
> 优化思路：按**业务边界**切分，每个模块独立维护，互不干扰。

---

## 当前文件的业务模块划分

```
edit.vue（1846 行）
├── 模块 A：运单信息展示          约 400 行模板 + 数据拍平逻辑
├── 模块 B：结算信息表单           SettlementInfo 组件（已独立）
├── 模块 C：费用明细               CostMessageDetails 组件（已独立）
├── 模块 D：风控预警               RiskWarning + RiskWarningDialog + provide/inject 回调
├── 模块 E：保存/提交流程          5 级调用链 + 油气校验 + 抹零 + 表单校验 + 风控
├── 模块 F：附件查看               handleLookImg + wabillStatus 分支映射
└── 模块 G：操作记录               getRecordPaging + EditMessageDetails
```

**维护成本高的根本原因**：A/D/E 三个模块的逻辑完全散落在 `edit.vue` 的 `data`/`methods`/`template` 各处，没有清晰边界，任何一处改动都要在 1800 行里定位。

## 方案一（优先）：保存/提交流程 — 5 级调用链拍平

### 现状

```
handleSave / handleSubmit
  └── changeWaybillStatus(status, type)
        └── [费用校验] → [油气校验] → [抹零计算] → [抹零弹窗]
              └── beforeChangeWaybill()
                    └── formRef.validate()
                          └── checkOrder()          ← .then() 风格
                                └── riskManagement()
                                      └── postData(busStatus)   ← 第一次存（保留当前状态）
                                            └── 风控测算
                                                  └── postSubmitAfterRiskPass()
                                                        └── postData('320')  ← 第二次存（真正提交）
```

**关键设计：隐性预保存**

`riskManagement` 里在风控测算**之前**会先调一次 `postData(busStatus)`，用当前状态把费用数据固化到后端，风控拿到的是已保存的最新数据。这不是 bug，是有意设计。

改提交流程时**必须保留这两次调用的顺序**，不能合并为一次，否则风控接口拿到的是旧数据。

**问题**：
- 新增一个前置校验（如"结算重量不能为0"）找不到准确的插入位置
- `postSubmitAfterRiskPass` 里用 `this.btnTypeClicked` 字符串判断是保存还是提交，这个标志位在 `handleSave`/`handleSubmit` 里设置，相隔约 200 行，极易遗漏
- 抹零弹窗通过 `$confirm` 回调继续调用 `beforeChangeWaybill`，中断后没有清理 `costNewList` 里已经 push 的抹零行，下次提交会重复计入
- `checkOrder` 用 `.then()/.catch()` 写法，而 `riskManagement` 用 `async/await`，链路里两种风格混用，`btnLoading` 的重置时机分散在不同位置
- `riskManagement` 开头有 `if (this.btnLoading) return` 防重，但 `checkOrder` 在调用它之前已经执行了 `this.btnLoading = false`，这个防重**永远不会触发**，是死代码

### 目标结构

```
handleSave()   → _executeFlow('save')
handleSubmit() → _executeFlow('submit')

async _executeFlow(action) {
  // 1. 费用明细校验（CostMessageDetails.commit()）
  // 2. 货损确认校验
  // 3. 油气费校验
  // 4. 抹零处理（返回 false 表示用户取消）
  // 5. 表单字段校验（formRef.validate）
  // 6. 运单状态校验（checkOrderStatus 接口）
  // 7. 风控测算（queryWaybillRiskInfoList）
  // 8. 实际提交（postData）
}
```

**好处**：
- 每个步骤是一个独立 async 方法，返回 `boolean` 表示是否继续，加/删/改某步骤只改 `_executeFlow` 里的一行
- `action` 直接传入，不需要 `btnTypeClicked` 标志位
- 抹零处理封装在 `_handleZeroMode()` 里，取消时在方法内部清理状态再返回 `false`

### 各步骤方法签名

```js
_executeFlow(action)          // 'save' | 'submit'
_validateCostDetails()        // → boolean，费用明细校验
_validateCargoLoss()          // → boolean，货损未确认时阻止
_validateOilGasCost()         // → boolean，油气费不超运费
_handleZeroMode()             // → Promise<boolean>，抹零处理（含弹窗确认）
_validateForm()               // → Promise<boolean>，formRef.validate
_checkOrderStatus()           // → Promise<boolean>，运单状态接口校验（含 warnTable 弹窗）
_runRiskAndSubmit(action)     // → 预保存 → 风控测算 → 有风险则弹窗/无风险则正式提交
```

> 注意：`_runRiskAndSubmit` 必须保留"预保存 → 风控"的两步顺序，不能合并为一次接口调用。
> 风控接口依赖后端已固化的费用数据，如果只调一次 postData('320') 跳过预保存，风控拿到的是旧数据。

### 抹零处理的关键修复

```js
async _handleZeroMode() {
  // 计算差值
  const diff = this._calcDiffAmount()
  if (!this.zeroByBusIdMode || diff === 0) return true

  // push 抹零行到 costNewList
  this._appendZeroRow(diff)

  // 等待用户确认
  const confirmed = await this._showZeroConfirmDialog(diff)
  if (!confirmed) {
    // 用户取消：移除已 push 的抹零行，避免下次重复计入
    this._removeZeroRow()
    return false
  }
  return true
}
```

**关联文件**：只改 `edit.vue`，不涉及其他文件。

**验证清单**：
```
□ 保存（310）：费用校验 → 油气校验 → 表单校验 → 接口，成功跳回列表
□ 提交（320）：全部校验 → 风控 → 接口，成功跳回列表
□ 任一校验失败：流程中止，按钮 loading 重置，用户可重试
□ 抹零弹窗取消：costNewList 不含抹零行，可重新保存
□ 低风险：跳过风控弹窗直接提交
□ 中高风险：弹窗 → 确认 → 提交；不通过 → 提示失败
```

## 方案二（优先）：CostMessageDetails — newList 双重初始化消除

### 现状

```
mounted()        → 初始化 newList（运费差价/货物保障/油气优惠三行）
detailObj watch  → 再次初始化 newList（逻辑相似但细节不同）
```

**问题**：
- `mounted` 的结果几乎立刻被 watch 覆盖，是"过渡态"
- 两处的油气判断代码几乎完全相同（50行重复）
- `mounted` 里全流程运单**不**强制 `premiumAmount` 为 0，watch 里**会**强制，不一致可能引发展示 bug
- `countMoney` 硬编码 `this.newList[0]` 取运费差价行，但 `companyType === 1` 时 `newList[0]` 是货物保障服务行，存在写错位置的风险

### 目标结构

```js
// 唯一初始化入口：detailObj watch
watch: {
  detailObj(val) {
    this.$nextTick(() => {
      if (this.newList.length === 0 || this.type !== 'edit') {
        this._buildNewList(val)      // 完整重建（含全流程处理）
      } else {
        this._updateOilGasOnly(val)  // 编辑模式保护：只更新油气行
      }
    })
  }
}

// mounted 里删除 newList 初始化，只做非业务的初始化
mounted() {
  window.addEventListener('resize', this.handleResize)
  const { type } = this.$route.query
  this.type = type
  this.busBillId = this.params.busBillId
  // newList 由 detailObj watch 负责，此处不处理
}
```

**countMoney 修复**：

```js
// 改前（脆弱）：
this.$set(this.newList[0], 'confirmMoney', serviceRateMoney)

// 改后（按名查找）：
const freightDiffRow = this.newList.find(i => i.costName === '运费差价')
if (freightDiffRow) this.$set(freightDiffRow, 'confirmMoney', serviceRateMoney)
```

**关联文件**：只改 `CostMessageDetails.vue`。

**验证清单**：
```
□ 进入 edit 页面：运费差价/货物保障/油气优惠行正常展示
□ 全流程运单：货物保障服务金额显示 0
□ companyType=1（B1）：newList 只有货物保障服务行，无运费差价行
□ 编辑确认金额后：运费差价联动更新（countMoney 触发）
□ 保存后刷新：newList 正确重建，不残留旧值
□ 修改确认金额（subtotal变化）→ 运费差价行金额同步更新
```

## 方案四：风控模块 — provide/inject 改为 emit

### 现状

```js
// edit.vue provide
provide() {
  return {
    updateRiskWarning:    this.updateRiskWarning,    // 子组件更新父组件 state
    handleCloseRiskDialog: this.handleCloseRiskDialog, // 子组件关闭时回调父组件
    handleSubmitRiskDialog: this.handleSubmitRiskDialog // 子组件提交时回调父组件
  }
}
```

**问题**：
- 子组件（`RiskWarningDialog`）通过 `inject` 反向调用父组件方法，数据流是**向上**的，不符合 Vue 单向数据流原则
- 追踪 `handleSubmitRiskDialog` 的触发时机需要跳到子组件里找 inject 的调用点
- `handleSubmitRiskDialog` 内有 `riskResult = null` 的变量声明但从未赋值，是无效代码

### 目标结构

```html
<!-- edit.vue 模板 -->
<RiskWarningDialog
  :visible.sync="riskWarningDialogVisible"
  :waybill-list="highMidWaybillList"
  @close="handleCloseRiskDialog"
  @submit="handleSubmitRiskDialog"
  @update="updateRiskWarning"
/>
```

```js
// RiskWarningDialog.vue 里把 inject 调用改为 emit
this.$emit('submit')
this.$emit('close')
this.$emit('update', info, type)
```

**关联文件**：`edit.vue` + `RiskWarningDialog.vue`（公共组件，改动前需确认 `index.vue` 等其他使用方是否兼容）。

> `warnTable` 的控制方式与风控相反（通过 `$refs` 直接调子组件方法），改方案四时一并统一为 emit 风格，避免两套不一致的调用模式共存。

**验证清单**：
```
□ 中高风险弹窗正常弹出
□ 点击关闭：什么都不做，弹窗关闭
□ 点击确认且 allowPass='100'：关闭弹窗，继续提交
□ 点击确认且 allowPass 不为 '100'：弹出失败提示
□ 单条处理完后无中高风险：自动关闭弹窗并提交
□ warnTable 弹窗展示/关闭正常
□ index.vue 使用 RiskWarningDialog 的场景不受影响
```

## 补充发现（代码走查追加）

以下是三份方案文档里未覆盖、走查代码后发现的问题，在推进各方案时需一并处理。

### 5.1 `riskManagement` 开头防重是死代码

```js
async riskManagement() {
  if (this.btnLoading) return  // ← 永远不会触发
  this.btnLoading = true
```

`checkOrder` 在调用 `riskManagement` 之前已经执行了 `this.btnLoading = false`，所以进入 `riskManagement` 时 `btnLoading` 始终为 false，这个防重无效。

**处理**：改方案一时，`_checkOrderStatus` 统一管理 loading 状态，删除 `riskManagement` 里的这行防重。

---

### 5.2 `handleLookImg` 中 wabillStatus 映射应提取为常量

```js
// 当前：3 层嵌套 if/else
if (transportTypeType === '3' || transportTypeType === '6') {
  if (btnType === '1') wabillStatus = '410'
  else if (btnType === '2') wabillStatus = '420'
  ...
} else if (...) { ... }
```

这段映射逻辑在 `edit.vue` 和 `read.vue` 里**完全相同**，是方案三里 430 行重复代码的一部分。

**处理**：提取为二维映射表，放进 `useWaybillDetail.js`：

```js
const WAYBILL_STATUS_MAP = {
  '3': { '1': '410', '2': '420', '3': '500' },
  '6': { '1': '410', '2': '420', '3': '500' },
  '1': { '1': '410', '2': '430', '3': '430' },
  '2': { '1': '410', '2': '430', '3': '430' },
  default: { '1': '400', '2': '420', '3': '500' }
}

handleLookImg(scope, btnType) {
  const map = WAYBILL_STATUS_MAP[scope.row.transportTypeType] || WAYBILL_STATUS_MAP.default
  const wabillStatus = map[btnType]
  // ... 后续调接口
}
```

---

### 5.3 `once` 标志位的设置时机不确定

```js
// getDetails 里设置 once = true
this.once = true
// updateOnce 里设置 once = false（在 $nextTick 里调用）
this.$nextTick(() => this.once = false)
```

`updateOnce` 在 `$nextTick` 里执行，但数据回填涉及多个异步步骤（`checkMemberStatus`、`getRiskDetails` 等），`$nextTick` 不能保证所有数据都已写入响应式属性，watch 可能在数据尚未完全回填时就触发联动，导致运费被提前覆盖。

**处理**：在方案三（`useWaybillDetail.js`）里统一封装 `getDetails`，确保 `once = false` 在所有异步步骤完成后才执行：

```js
async getDetails() {
  this.once = true
  // ... 所有数据回填
  await this.checkMemberStatus(...)
  await this.getRiskDetails()
  // 全部完成后再开放联动
  this.$nextTick(() => { this.once = false })
}
```

---

### 5.4 `handleSubmitRiskDialog` 里的无效变量

```js
async handleSubmitRiskDialog() {
  let canSubmit = false
  const riskResult = null   // ← 声明后从未赋值，始终为 null
  if (this.highMidWaybillList[0].allowPass !== '100') {
    canSubmit = false
  } else {
    canSubmit = true
  }
  if (!canSubmit) {
    // riskResult 在这里被判断但永远是 null
    if (riskResult) {
      content.push(h('p', null, '失败原因：【运单运价异常】...'))
    } else {
      content.push(h('p', null, '失败原因：存在风险无法操作...'))
    }
  }
}
```

`riskResult` 恒为 `null`，导致失败提示永远走 `else` 分支，更详细的失败原因永远不会展示。

**处理**：改方案四时，确认是否有后续需要填充 `riskResult` 的计划，如果没有则直接删除这个变量和相关 `if` 分支，只保留通用提示。

## 执行路线图

### 阶段 1 — 独立可验证，互不依赖（预估各 0.5~1d）

**方案二：CostMessageDetails 双重初始化消除**
- 删除 `mounted` 里的 `newList` 初始化，统一到 `detailObj` watch
- 提取 `_buildNewList` / `_updateOilGasOnly` 两个明确命名的方法
- 修复 `countMoney` 硬编码 `newList[0]` 问题（改为 `find` 按名查找）

**方案一：保存/提交流程拍平**
- 提取 `_executeFlow(action)` 作为统一入口
- 提取各校验步骤为独立方法（含 5.1 里的死代码清除）
- 修复抹零取消后 `costNewList` 未清理的问题
- 保留"预保存 → 风控 → 正式提交"的两次接口调用顺序（见 5.1 说明）
- 顺带处理 5.4 里 `riskResult = null` 的无效变量

### 阶段 2 — 依赖阶段 1（预估 0.5d）

**方案四：风控 + warnTable 通信方式统一**
- `RiskWarningDialog`：`provide/inject` 改为 `$emit`（改前确认 `index.vue` 等其他使用方的兼容性）
- `warnTable`：`$refs.warnTable.handleOpenDailog()` 改为 `:visible` prop + `@confirm` emit，与风控保持一致风格
- 顺带处理 5.4 中已确认无用的提示分支

### 阶段 3 — 体量最大，独立排期（预估 2d）

**方案三：useWaybillDetail mixin + WaybillInfoTable 组件**
- 新建 `useWaybillDetail.js`，迁入 `getDetails` / `handleLookImg` / `getRecordPaging` / `loadStatusOption` / `handleViewCarInfo`
- 提取 5.2 里的 `WAYBILL_STATUS_MAP` 常量（消除 `handleLookImg` 里的 3 层嵌套 if/else）
- 修复 5.3 里 `once` 标志位设置时机：确保 `once = false` 在所有异步步骤完成后执行
- 新建 `WaybillInfoTable.vue`，剥离运单信息表格模板
- `edit.vue` 和 `read.vue` 分别引入 mixin 和组件

**不建议同时推进多个阶段**，每个阶段做完后完整测试一遍再启动下一个。
