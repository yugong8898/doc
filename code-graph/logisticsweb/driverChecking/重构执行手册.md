## 第一波：修 Bug（用户直接受影响）

### T1 — $total 重复调用导致汇总行叠加

**文件**：`add.format.js`

**现状**：`$total()` 先 `pop()` 一项再 `push()` 追加多行。若当前已有多个汇总行，`pop()` 只移除最后一行，重复调用时小计/合计行越积越多。用户在详情页切换 tab 返回后，看到 2 行小计、2 行合计。

**目标**：`$total()` 幂等——无论调用几次，费用表格始终只有 1 行小计、1 行合计。

**改法**：
```js
$total() {
  // 先移除所有已有特殊行，保证幂等
  this.reseponse.model.bmsBusinessCostList =
    this.reseponse.model.bmsBusinessCostList.filter(item => !item.specialRowType)
  if (this.reseponse.model.bmsBusinessCostList.length < 2) return
  // ... 原有追加逻辑不变
}
```

**测试重点**：

| 场景 | 预期 |
|------|------|
| 进入详情页（read/verify/check） | 小计行和合计行各只有 1 行 |
| 切换浏览器 tab 再切回来 | 汇总行不重复 |
| 全流程运单（fullProcessFlag=10） | 油气优惠 + fpDiff + shipperPay/carrierPay 各 1 行 |
| companyType=1（B1） | 不显示运费差价行和合计行 |

**边界情况**：
- `bmsBusinessCostList` 为空或只有 1 项 → 不追加任何汇总行
- 后端返回的数据本身已带 `specialRowType` 标记 → filter 会误删 → 需确认后端不会返回此字段

**收益**：修复线上真实 bug，用户不再看到重复的汇总行。

### T2 — codeFormat 无匹配时列表显示空白

**文件**：`index.format.js`

**现状**：`codeFormat` 方法的 switch 没有命中任何 case 时，`options` 未定义，`options?.find()` 返回 `undefined`，列表单元格渲染为空白。

**目标**：任何情况下 `codeFormat` 都返回有意义的字符串，无匹配时显示 `--`。

**改法**：
```js
codeFormat(prop, code) {
  let options
  switch (prop) {
    // ... 各 case 不变
  }
  const option = options?.find(item => item.code === (Number.isNaN(+code) ? code : +code))
  return option ? option.text : '--'
}
```

**测试重点**：

| 场景 | 预期 |
|------|------|
| 正常数据（运输类型=1/4/5） | 显示"公路"/"铁路"/"水路" |
| 后端返回未知 code（如 99） | 显示 `--` 而非空白 |
| code 为 null/undefined | 显示 `--` |

**边界情况**：
- `prop` 传入未定义的类型名 → `options` 为 undefined → `options?.find()` 安全返回 undefined → 兜底 `--`
- `code` 为字符串 `'1'` 而非数字 `1` → `Number.isNaN(+code)` 处理了类型转换

**收益**：列表页不再出现空白单元格，数据展示完整。

## 第四波：核心方法重构（可维护性）

### T13 — $total 拆解为子方法

**文件**：`add.format.js`

**依赖**：T1 完成后执行（T1 已加 filter 保证幂等）

**现状**：`$total` 方法体约 80 行，小计/运费差价/货物保障/油气优惠/合计/全流程专属行的构建逻辑全堆在一起。任何一种行的条件变化都要在 80 行中定位，新增一种特殊行需要通读全部逻辑。

**目标**：每种特殊行有独立的构建方法，`$total` 主体精简为"过滤 → 计算 → 按条件追加"的线性调用序列。新增/删除某种行只需加减一行调用。

**改法**：
```js
// 提取子方法
_calcSubtotal(list)          // 纯计算，返回小计数据，不操作数组
_appendSubtotalRow(data)     // 追加小计行
_appendFreightDiffRow()      // 运费差价行（companyType !== 1 时）
_appendInsuranceRow()        // 货物保障服务行（premiumBuyerType === '10'）
_appendOilDiscountRow()      // 油气优惠行
_appendTotalRow()            // 合计行（companyType !== 1 时）
_appendFullProcessRows()     // 全流程专属行（fpDiff / shipperPay / carrierPay）

// $total 主体
$total() {
  this.reseponse.model.bmsBusinessCostList =
    this.reseponse.model.bmsBusinessCostList.filter(item => !item.specialRowType)
  if (this.reseponse.model.bmsBusinessCostList.length < 2) return
  const data = this._calcSubtotal(this.reseponse.model.bmsBusinessCostList)
  this._appendSubtotalRow(data)
  if ((action === 'check' || action === 'read') && freightType === '2') {
    this._appendFreightDiffRow()
    this._appendInsuranceRow()
    this._appendOilDiscountRow()
    this._appendTotalRow()
    this._appendFullProcessRows()
  }
}
```

**测试重点**：

| 场景 | 预期 |
|------|------|
| 普通运单（freightType='1'） | 只有小计行 |
| 网货运单 verify action | 只有小计行（verify 不追加其他行） |
| 网货运单 read/check action | 小计 + 运费差价 + 合计 |
| 全流程运单 read/check | 小计 + 运费差价 + 油气优惠 + 合计 + fpDiff + shipperPay/carrierPay |
| companyType=1（B1） | 无运费差价行、无合计行 |
| premiumBuyerType='10' | 有货物保障服务行 |
| premiumBuyerType≠'10' | 无货物保障服务行 |

**边界情况**：
- `bmsBusinessCostList` 为空 → length < 2 直接 return，不追加任何行
- 油气数据无效（oilAmount=0 且 oilDiscountAmount=-1）→ `_appendOilDiscountRow` 内部判断不追加
- 全流程运单但 companyType 既非 1 也非 2 → 不追加 shipperPay/carrierPay

**收益**：
- 每种特殊行的逻辑独立维护，改一种不影响其他
- 新增特殊行只需写一个 `_appendXxxRow` 方法 + 在 `$total` 里加一行调用
- 单独测试某一种行的条件成为可能

### T14 — $action switch/case 改为对象映射

**文件**：`index.format.js`

**依赖**：T11 完成后执行（常量已引入）

**现状**：`$action` 用 switch/case 列出所有状态约 40 行，8 个状态 × 6 种操作，逻辑散落各处，新增状态需找到所有相关 case。

**目标**：用一张二维映射表描述「状态 → 操作」的关系，一眼可查全貌。

**改法**：
```js
import { BUS_STATUS } from './constants'

const ACTION_MAP = {
  [BUS_STATUS.PENDING_VERIFY]: { read: true,  verify: true,  check: false, giveBack: false, bill: false, affirm: false },
  [BUS_STATUS.PENDING_CHECK]:  { read: true,  verify: false, check: true,  giveBack: false, bill: false, affirm: false },
  [BUS_STATUS.PROCESSING]:     { read: true,  verify: false, check: false, giveBack: false, bill: false, affirm: false },
  [BUS_STATUS.CHECK_FAILED]:   { read: true,  verify: false, check: false, giveBack: false, bill: false, affirm: false },
  [BUS_STATUS.CHECK_PASSED]:   { read: true,  verify: false, check: false, giveBack: true,  bill: true,  affirm: false },
  [BUS_STATUS.PENDING_AFFIRM]: { read: true,  verify: false, check: false, giveBack: true,  bill: false, affirm: true  },
  [BUS_STATUS.AFFIRM_BACK]:    { read: true,  verify: false, check: false, giveBack: false, bill: false, affirm: false },
  [BUS_STATUS.AFFIRM_PASSED]:  { read: true,  verify: false, check: false, giveBack: false, bill: false, affirm: false },
}
const DEFAULT_CTRL = { read: false, verify: false, check: false, giveBack: false, bill: false, affirm: false }

$action(row, action) {
  return (ACTION_MAP[+row.busStatus] || DEFAULT_CTRL)[action]
}
```

**测试重点**：

| 场景 | 预期 |
|------|------|
| 待核对(100) | 显示：查看 + 核对 |
| 待审核(110) | 显示：查看 + 审核 |
| 处理中(115) | 只显示：查看 |
| 审核不通过(120) | 只显示：查看 |
| 审核通过(130) | 显示：查看 + 退回 + 推送 |
| 待对账(200) | 显示：查看 + 退回 + 对账确认 |
| 对账退回(210) | 只显示：查看 |
| 对账通过(220) | 只显示：查看 |
| 未知状态(999) | 全部不显示（DEFAULT_CTRL） |

**边界情况**：
- `row.busStatus` 为字符串 `'100'` → `+row.busStatus` 转为数字 100 → 命中映射
- `row.busStatus` 为 null/undefined → `+null = 0` → 命中 DEFAULT_CTRL → 全部 false
- `action` 传入未定义的操作名 → 返回 undefined（falsy） → 按钮不显示

**收益**：
- 状态与操作的映射关系一目了然，不需要在 40 行 switch 里追踪
- 新增状态只需在 ACTION_MAP 里加一行
- 新增操作类型只需在每个状态对象里加一个字段

## 第五波：组件化（长期维护收益）

### T15 — 费用表格抽为独立组件 DriverCheckingCostTable.vue

**文件**：新建 `driverChecking/components/DriverCheckingCostTable.vue` + 改动 `add.vue`

**依赖**：T13 完成后执行（$total 已拆好子方法，逻辑稳定）

**现状**：`add.vue` 费用表格模板全部内联（约 200 行），read/verify/check 三种模式的条件判断混在一起。表格列需要调整时（比如新增一列、改列宽、改显隐条件），只能在 `add.vue` 这个大文件里定位修改，无法单独测试费用表格的展示逻辑。

**目标**：费用表格独立为 `DriverCheckingCostTable.vue`，`add.vue` 只需一行 `<DriverCheckingCostTable ... />` 引入。表格列的调整只改组件一处。`add.vue` 行数从约 500+ 行模板降低约 200 行。

**组件接口设计**：
```
Props:
  action         String   'read' / 'verify' / 'check'
  costList       Array    bmsBusinessCostList（含特殊行）
  partPayData    Object   分次付款数据（预付/回单付/到付）
  showStar       Boolean  是否脱敏展示金额
  dictionary     Object   枚举字典（费用类型等）

Emits:
  cost-change    { index, field, value }   确认金额变更
  files-change   { index, files }          附件变更
```

**add.vue 模板替换**：
```html
<DriverCheckingCostTable
  :action="$route.params.action"
  :cost-list="reseponse.model.bmsBusinessCostList"
  :part-pay-data="partPayData"
  :show-star="showStar"
  :dictionary="dictionary"
  @cost-change="handleCostChange"
  @files-change="handleFilesChange"
/>
```

**测试重点**：

| 场景 | 预期 |
|------|------|
| read action | 表格全部只读，金额列正常展示 |
| verify action | 确认金额列可编辑，修改后 emit cost-change |
| verify action 附件上传 | 上传成功后 emit files-change |
| check action | 表格只读，显示审核结果列 |
| 小计行/合计行 | 特殊样式（合并单元格/加粗） |
| showStar=true | 金额列展示星号脱敏 |
| costList 为空 | 表格展示空状态 |

**边界情况**：
- `partPayData` 接口未返回时为空对象 → 组件内做空值兜底，不展示分次付款区域
- 特殊行（specialRowType）的列合并逻辑 → 组件内通过 `span-method` 处理
- verify 模式下用户清空确认金额 → emit 出去的 value 为空字符串，父组件需决定是否允许

**收益**：
- 费用表格逻辑内聚，改列/改样式只改一个组件文件
- `add.vue` 减少约 200 行模板代码
- 后续如果其他模块需要类似费用表格，可直接复用
