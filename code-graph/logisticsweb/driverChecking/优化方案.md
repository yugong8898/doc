# 司机对账模块 — 优化方案

> 目标：方便后续功能迭代和维护，局部渐进式推进，不做一刀切的大重构。

---

## 总体原则

- **P0（随时可做）**：改动不影响业务逻辑，可在日常迭代中顺手处理，30min 内能完成
- **P1（触碰时做）**：下次需求触碰到该代码时顺手重构，不单独排期
- **P2（专项排期）**：涉及组件拆分/通信调整，需单独排期 + 充分测试

## P1 — $total 方法拆分

### 任务 5：将 add.format.js 的 $total 拆成职责单一的子方法

**文件**：`src/views/userBase/driverChecking/add.format.js`

**问题**：`$total()` 约 190 行，混合了"计算小计"和"按条件追加 6-8 种特殊汇总行"两件事。新增/修改任意一种汇总行都要通读全文。

**触发时机**：下次需要新增汇总行（如新增字段、调整油气优惠逻辑）时执行。

**目标结构**：

```js
$total() {
  if (this.reseponse.model.bmsBusinessCostList.length < 2) return
  const subtotalData = this._calcSubtotal()           // 纯计算，不修改数组
  this._appendSubtotalRow(subtotalData)               // 追加小计行

  const isNetFreight = this.$route.params.action !== 'verify'
    && this.reseponse.model.freightType === '2'
  if (!isNetFreight) return

  if (this.reseponse.model.companyType !== 1) {
    this._appendFreightDiffRow()                      // 运费差价
  }
  if (this.reseponse.model.premiumBuyerType === '10') {
    this._appendInsuranceRow()                        // 货物保障服务
  }
  this._appendOilDiscountRow()                        // 油气优惠（含全流程强制为0）

  if (this.reseponse.model.companyType !== 1) {
    this._appendTotalRow(subtotalData)                // 合计
  }
  if (this.reseponse.model.fullProcessFlag === 10) {
    this._appendFullProcessRows()                     // 全流程三行
  }
},

// ---- 子方法 ----

// 纯函数：计算小计，返回数据对象，不直接操作列表
_calcSubtotal() {
  const parsePrice = window.parsePrice
  const parsePriceReduction = window.parsePriceReduction
  this.reseponse.model.bmsBusinessCostList.pop()
  const list = JSON.parse(JSON.stringify(this.reseponse.model.bmsBusinessCostList))
  return list.reduce((acc, item) => {
    ['busCost', 'updBusCost'].forEach(k => { item[k] = parsePriceReduction(item[k]) })
    const isAdd = ['200', '600'].includes(`${item.costType}`)
    return {
      costName: isAdd
        ? new BigNumber(acc.costName).plus(item.busCost || 0).toFixed(2)
        : new BigNumber(acc.costName).minus(item.busCost || 0).toFixed(2),
      $adjustCost: isAdd
        ? new BigNumber(acc.$adjustCost).plus(item.updBusCost || 0).toFixed(2)
        : new BigNumber(acc.$adjustCost).minus(item.updBusCost || 0).toFixed(2)
    }
  }, { costName: 0, $adjustCost: 0 })
},

_appendSubtotalRow(data) {
  this.reseponse.model.bmsBusinessCostList.push({
    $adjustCost: window.parsePrice(window.parsePriceReduction(data.$adjustCost)),
    costType: { count: true, num: window.parsePrice(window.parsePriceReduction(data.costName)) },
    count: true,
    specialRowType: 'subtotal'
  })
},

_appendFreightDiffRow() {
  this.reseponse.model.bmsBusinessCostList.push({
    costType: '', count: true, specialRowType: 'freightDiff',
    updBusCost: this.showStar
      ? '差价率: *%'
      : `差价率: ${this.reseponse.model.taxRate || 0}%`
  })
},

_appendInsuranceRow() {
  this.reseponse.model.bmsBusinessCostList.push({
    costType: '', count: true, specialRowType: 'insurance'
  })
},

_appendOilDiscountRow() {
  let { oilAmount, oilDiscountAmount, fullProcessFlag } = this.reseponse.model
  const hasValidOilData = oilAmount > 0
    && oilDiscountAmount !== -1 && oilDiscountAmount !== '-1'
    && Number(oilDiscountAmount) !== -1

  if (fullProcessFlag === 10) {
    // 全流程：强制展示，金额为 0
    oilAmount = 0
    oilDiscountAmount = 0
  } else if (!hasValidOilData) {
    return
  }
  this.reseponse.model.bmsBusinessCostList.push({
    costType: '', count: true, specialRowType: 'oilDiscount',
    oilAmount,
    oilDiscountAmount: oilDiscountAmount || 0,
    updBusCost: `油气实际使用 ${window.parsePrice(oilAmount)} 元`
  })
},

_appendTotalRow() {
  this.reseponse.model.bmsBusinessCostList.push({
    costType: '', count: true, specialRowType: 'total'
  })
},

_appendFullProcessRows() {
  const m = this.reseponse.model
  // 全流程运费差价
  this.reseponse.model.bmsBusinessCostList.push({
    costType: '', count: true, specialRowType: 'fpDiff',
    updBusCost: `差价率：${BigNumber(m.fpFreightDiffRate || 0).times(100).toFixed(2)}%`
  })
  // 客户应付运费（B1公司）
  if (m.companyType === 2) {
    this.reseponse.model.bmsBusinessCostList.push({
      costType: '', count: true, specialRowType: 'shipperPay',
      updBusCost: `上游客户名称：${m.shipperCompanyName || ''}`
    })
  }
  // 应付承运商运费（B2公司）
  if (m.companyType === 1) {
    this.reseponse.model.bmsBusinessCostList.push({
      costType: '', count: true, specialRowType: 'carrierPay',
      updBusCost: `承运商名称：${m.carrierCompanyName || ''}`
    })
  }
}
```

**验证清单**：

```
□ 查看页（read）：小计 / 运费差价 / 货物保障 / 油气优惠 / 合计 展示正常
□ 审核页（check）：同上
□ 核对页（verify）：只显示最后一行小计，无其他汇总行
□ 全流程运单（fullProcessFlag=10）：全流程差价 / 客户应付 / 应付承运商 展示正常
□ companyType=1 运单：运费差价行 / 合计行 不显示
□ premiumBuyerType='20' 运单：货物保障服务行不显示
□ 油气优惠 oilDiscountAmount=-1 时：油气优惠行不显示（非全流程）
□ ESLint 通过，无报错
```

## P1 — $action 改为数据驱动

### 任务 6：将 index.format.js 的 $action switch/case 改为映射表

**文件**：`src/views/userBase/driverChecking/index.format.js`

**问题**：`$action` 用 switch/case 列举 8 个状态，新增状态需要手动加 case，容易遗漏，阅读时也需要从多个 case 里拼凑一个状态的完整权限。

**触发时机**：下次新增状态或修改按钮规则时执行。

**目标结构**：

```js
import { BUS_STATUS } from './constants'

// 状态 → 可用操作的映射表（一眼看出每个状态开了哪些操作）
const ACTION_MAP = {
  [BUS_STATUS.PENDING_VERIFY]: { read: true, verify: true },
  [BUS_STATUS.PENDING_CHECK]:  { read: true, check: true },
  [BUS_STATUS.PROCESSING]:     { read: true },
  [BUS_STATUS.CHECK_FAILED]:   { read: true },
  [BUS_STATUS.CHECK_PASSED]:   { read: true, giveBack: true, bill: true },
  [BUS_STATUS.PENDING_AFFIRM]: { read: true, giveBack: true, affirm: true },
  [BUS_STATUS.AFFIRM_BACK]:    { read: true },
  [BUS_STATUS.AFFIRM_PASSED]:  { read: true }
}

$action(row, action) {
  const controls = ACTION_MAP[+row.busStatus] || {}
  return !!controls[action]
}
```

**验证清单**：

```
□ 待核对（100）：只显示查看、核对按钮
□ 待审核（110）：只显示查看、审核按钮
□ 审核通过（130）：显示查看、退回、推送按钮
□ 待对账（200）：显示查看、退回、对账确认按钮
□ 其他状态：只显示查看按钮
```

## P2 — 费用表格独立为组件

### 任务 7：将 add.vue 的费用表格抽为 DriverCheckingCostTable 组件

**新建文件**：`src/views/userBase/driverChecking/components/DriverCheckingCostTable.vue`
**改动文件**：`src/views/userBase/driverChecking/add.vue`（模板减少约 250 行）

**问题**：`add.vue` 模板超 500 行，费用表格内嵌了大量 `v-if="$route.params.action === 'verify'"` 分支。三种 action 模式的渲染逻辑混在一起，新增一列需要通读三套分支才敢改。

**触发时机**：下次费用表格有需求变更时（新增费用类型、调整列配置等）专项排期。

**组件接口设计**：

```
DriverCheckingCostTable
  props:
    action        String   'read' | 'verify' | 'check'
    costList      Array    bmsBusinessCostList（含特殊汇总行）
    partPayData   Array    分次支付数据（小计行 tooltip 用）
    showStar      Boolean  是否脱敏展示
    dictionary    Object   枚举字典（费用类型映射）

  emits:
    cost-change(index, field, value)  确认金额 / 备注变化
    files-change(index, files)        附件变化
```

**拆分步骤**（每步独立可测）：

1. 新建 `DriverCheckingCostTable.vue`，把 `add.vue` 费用表格的模板代码剪切进去，先不改逻辑
2. 把 `add.format.js` 中的 `spanMethod`、`formItemRules`、`formatTableFilse` 移入组件内部
3. 将任务 5 拆分好的子方法也移入组件内部（`mounted` 时调 `$total`）
4. 用 `emit('cost-change')` 替代直接修改 `reseponse.model`
5. `add.vue` 替换为 `<DriverCheckingCostTable v-bind="costTableProps" @cost-change="handleCostChange" />`

**验证清单**：

```
□ verify 模式：确认金额列可输入，输入后小计联动更新
□ read / check 模式：全部只读，特殊汇总行正常展示
□ 附件查看 / 上传正常
□ 特殊行合并单元格（spanMethod）正常
□ 提交审核时表单校验（updBusCost 必填）正常触发
□ 油气优惠行 / 全流程行在不同 companyType 下正确显示 / 隐藏
□ ESLint 通过
```
