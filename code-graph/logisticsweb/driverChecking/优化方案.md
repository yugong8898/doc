# 司机对账模块 — 优化方案

> 目标：方便后续功能迭代和维护，局部渐进式推进，不做一刀切的大重构。
>
> 优先级说明：**P0** 随时可做 / **P1** 触碰该文件时顺手做 / **P2** 需单独排期+测试

---

## 一、问题全景（按维度分析）

### 1.1 可读性问题

| 位置 | 问题描述 |
|------|---------|
| `index.format.js` `$action` | switch/case 8个分支，每个分支都展开写完整对象 `{ read:true, verify:false, ... }`，新增一个操作需要改8处 |
| `add.format.js` `$total` | 190行，混合"计算小计"和"追加6种汇总行"两件事，读懂需要通读全文 |
| `add.format.js` `getTransportName` | `case 1/2/3` 三个 case 返回同一个值 `'highway'`，应合并 |
| `index.conf.js` `buttonAct` | 整段注释掉但无废弃原因说明，不敢删除 |
| `add.format.js` `$otherKuiTonsRatio` | `return` 后跟大段注释掉的旧逻辑，制造阅读干扰 |
| `index.format.js` `$all` | 末尾有行注释掉的旧计算逻辑，原因不明 |
| `add.format.js` `codeFormat` | `reviewResult` 和 `rReviewResult` 两个 case 返回同一份数组，重复定义 |
| `add.format.js` `formatTableFilse` / `formatResultFilse` | 两个方法实现完全相同，只有名字不同 |

### 1.2 魔法数字问题

| 位置 | 裸数字/字符串 | 含义 |
|------|------------|------|
| `index.format.js` `$action` | `100/110/115/120/130/200/210/220` | 账单各状态码 |
| `add.format.js` `$total` | `'200'/'600'` | 应付/其他应付费用类型 |
| `add.format.js` `$total` | `'check'/'read'/'verify'` | action 模式标识 |
| `add.format.js` `$total` | `'2'` (freightType) | 网络货运 |
| `add.format.js` `$otherKuiTonsRatio` | `[1, 2]` (transportType) | 公路运输类型 |
| `index.vue` `batchAffirm` | `['110','130','200']` | 批量确认允许状态 |
| `index.vue` `handleBatchDisPass` | `['110']` | 批量不通过允许状态 |

### 1.3 结构问题

| 位置 | 问题描述 |
|------|---------|
| `add.format.js` `$total` | 每种汇总行的追加逻辑内联在同一个方法里，新增一种汇总行需要通读全文找到插入位置 |
| `index.conf.js` `form.list` | 搜索区字段全部定义在 conf 里，但 `index.vue` 模板里又用 `v-for + v-else-if` 重新处理每一个字段，conf 和模板双重维护，新增字段要改两处 |
| `add.vue` 运单信息区 | `conf.formList` 里定义字段，模板里 `v-else-if` 分支逐个处理 slot，新增字段要改 conf 和模板两处，且 `$otherKuiTonsRatio` 在模板里调用了两次（判断颜色+展示文字） |

### 1.4 数据流问题

| 位置 | 问题描述 |
|------|---------|
| `add.format.js` `$total` | 直接 `.pop()` 和 `.push()` 操作 `this.reseponse.model.bmsBusinessCostList`，有副作用，函数不纯，重复调用会出错 |
| `add.format.js` `fetchOilGasConfig` | 接口调用结果仅用于写 `oilGasConfig`，但 `$total` 里的油气行显示条件判断与 `oilGasConfig` 无关，两者实际解耦，`fetchOilGasConfig` 的存在意义需确认 |

### 1.5 健壮性问题

| 位置 | 问题描述 |
|------|---------|
| `index.format.js` `codeFormat` | `enumensFormat` 为 null 时直接 return undefined（无 return 语句），调用方拿到 undefined 会渲染空白而不是 `'--'` |
| `add.format.js` `$total` | `bmsBusinessCostList.length < 2` 时直接 return，但外部调用前没有保障列表已初始化，边界条件需靠调用方保证 |

| # | 文件 | 问题 | 操作 | 状态 |
|---|------|------|------|------|
| 8 | `index.format.js` | `$action` switch/case 可读性差，新增状态要改多处 | 改为 `ACTION_MAP` 对象映射，方法体缩为 2 行 | ⬜ |

| 12 | `add.format.js` | `$total` 直接 `.pop()` 操作响应式数组，有副作用 | 在 `_calcSubtotal` 里先 pop，标注只能在 `$total` 开头调用一次 | ⬜ |

**`$action` 改造目标**：
```js
// 改前：8个 case，每个写完整对象
// 改后：一张表，一行代码
const ACTION_MAP = {
  [BUS_STATUS.PENDING_VERIFY]: { read: true, verify: true },
  [BUS_STATUS.PENDING_CHECK]:  { read: true, check: true },
  [BUS_STATUS.PROCESSING]:     { read: true },
  [BUS_STATUS.CHECK_FAILED]:   { read: true },
  [BUS_STATUS.CHECK_PASSED]:   { read: true, giveBack: true, bill: true },
  [BUS_STATUS.PENDING_AFFIRM]: { read: true, giveBack: true, affirm: true },
  [BUS_STATUS.AFFIRM_BACK]:    { read: true },
  [BUS_STATUS.AFFIRM_PASSED]:  { read: true }
}
$action(row, action) {
  return !!(ACTION_MAP[+row.busStatus] || {})[action]
}
```

**`$total` 拆分后结构**：
```
$total()
  ├── _calcSubtotal()              → 纯计算，返回 { costName, $adjustCost }
  ├── _appendSubtotalRow(data)     → push 小计行
  ├── _appendFreightDiffRow()      → push 运费差价行（companyType !== 1）
  ├── _appendInsuranceRow()        → push 货物保障服务行（premiumBuyerType === '10'）
  ├── _appendOilDiscountRow()      → push 油气优惠行（含全流程强制为0逻辑）
  ├── _appendTotalRow()            → push 合计行（companyType !== 1）
  └── _appendFullProcessRows()     → push 全流程差价/客户应付/应付承运商三行
```
