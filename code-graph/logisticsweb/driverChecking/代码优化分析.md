# 司机对账 — 代码优化分析

> 涉及文件：`index.vue` / `add.vue` / `index.conf.js` / `index.format.js` / `add.conf.js` / `add.format.js`
> 技术栈：Vue2 + JS + Element UI（logisticsweb）
> 目标：在不改变外部行为的前提下，提升可读性、可维护性，降低后续迭代成本

---

## 二、`index.vue` 问题

### 问题 1：搜索区模板 ~220 行 `v-else-if` 平铺

每个搜索字段都是一个独立的 `<template v-else-if="col.prop === 'xxx'">` 分支，共 ~22 个分支。新增或修改字段需同时改：`searchFormColumns` 数组 + 模板分支 + `filterModel` 初始值，三处容易遗漏。

与 `waybillSettlement/index.vue` 的搜索区结构高度相似，但实现细节不同（`filterModel` vs `searchForm`，字段数量也略有差异），两个模块分别维护，无法共用。

**优化方向**：将字段渲染配置（`type`、`placeholder`、`options`）收拢到 `searchFormColumns` 里，模板改为统一渲染器，字段增减只改配置，不动模板。

```javascript

```

中间有整行把十几个字段堆在一起，是历史迭代追加时图省事的结果，可读性极差，字段业务含义完全靠记忆。

**优化方向**：每个字段单独一行，非显然字段加上业务含义注释，零风险。

### 问题 4：`getDefaultMainId` / `networkMainBodyFilter` 与 `waybillSettlement/index.vue` 完全重复

```javascript
// driverChecking/index.vue 和 waybillSettlement/index.vue 里完全相同
getDefaultMainId(companyId) {
  getNewCompanyInfos({ companyId, socialCreditCode: '', mainType: '2' }).then(response => {
    if (response.model && response.model.length > 0) {
      this.networkMainBodyList = response.model
      this.filterNetworkMainBodyList = response.model
    }
  })
},
networkMainBodyFilter(val) {
  if (val) {
    this.filterNetworkMainBodyList = this.networkMainBodyList.filter(item =>
      item.mainName.toLowerCase().startsWith(val.toLowerCase())
    )
  } else {
    this.filterNetworkMainBodyList = this.networkMainBodyList
  }
},
```

**优化方向**：提取为 `networkMainBodyMixin.js`，两个模块共用。

### 问题 5：`handleStar` 手机号打星方法在 index/add/waybillSettlement 多个文件重复

```javascript
// 多个文件里完全相同
handleStar(content, showFlagSplit) {
  if (typeof content !== 'string' || content.length < 7) return
  return !showFlagSplit
    ? content.slice(0, 3) + '*'.repeat(content.length - 7) + content.slice(-4)
    : content
}
```

**优化方向**：迁入 `unitTool.js` mixin 或 `@/utils/format.js`，统一命名为 `maskSensitive`，各处调用改为引用。

### 问题 6：`getTime` / `buildTime` 日期工具函数内联

```javascript
// index.vue 内联了一个 buildTime 局部函数
getTime() {
  function buildTime(time) {
    var date = new Date(time)
    var Y = date.getFullYear() + '-'
    var M = (date.getMonth() + 1 < 10 ? '0' + (date.getMonth() + 1) : date.getMonth() + 1) + '-'
    var D = (date.getDate() < 10 ? '0' + date.getDate() : date.getDate())
    return Y + M + D
  }
  // ...
}
```

项目里已有 `@/utils/func.js`，这种格式化函数应该放在那里。

**优化方向**：将 `buildTime` 迁入 `@/utils/func.js`，方法名改为 `formatDateToYMD`。

---

### 问题 7：`formatLongText` / `cutLongText` 两个相似工具方法职责不清晰，且只在 `add.vue` 使用

```javascript
// index.vue 里定义，但逻辑上只用于 add.vue 展示的文本格式化
formatLongText(text) { /* 每 50 字换行 */ },
cutLongText(text) { /* 超 20 字截断 + ... */ },
```

这两个方法定义在 `index.vue` 里但从使用场景看更应该在 `add.vue` 或 `add.format.js`。

**优化方向**：迁移到实际使用方，或者迁入 `@/utils/format.js`。

---

### 问题 8：`console.log` 调试输出残留

```javascript
// index.vue computed funcId
funcId() {
  console.log(this.$route.meta.funcId, 'this.$route.meta.funcId')
  return String(this.$route.meta.funcId)
},
```

`computed` 里的 `console.log` 会在每次依赖变化时触发，频繁输出。

**优化方向**：直接删除。

## 三、`add.vue` 问题

### 问题 9：`conf.js` + `format.js` 拆文件方式增加了认知成本

`add.vue` 依赖 `add.conf.js` 和 `add.format.js` 两个 mixin，三个文件高度耦合（互相引用 `this`），但对不熟悉这套约定的人来说，需要在三个文件之间跳来跳去才能理解一个字段的完整逻辑。

**优化方向（不建议现阶段改架构）**：在 `add.conf.js` 顶部补说明注释，标注依赖关系和使用约定；在各字段配置旁补充业务含义注释，降低后续维护理解成本。参考格式：

```javascript
/**
 * add.conf.js — 司机对账详情页配置（作为 add.vue 的 mixin 使用）
 * 依赖：add.vue 的 data/methods（通过 this 访问）
 * 依赖：add.format.js 提供的计算方法
 * 注意：不要在此文件中直接访问 DOM 或 $refs
 */
```

---

### 问题 10：`add.vue` 中费用汇总行由 `add.format.js` 的 `$total()` 动态追加，维护难度高

汇总行通过 `specialRowType` 字段标识，逻辑散落在：
- `add.format.js`：追加汇总行数据
- `add.conf.js`：`spanMethod` 控制合并列
- `add.vue` 模板：按 `specialRowType` 渲染不同样式

三处联动，任何一处改动都需要同步其他两处，且没有文档说明特殊行的数据结构。

**优化方向**：在 `add.format.js` 的 `$total()` 方法顶部补充一段完整的数据结构注释，说明每种 `specialRowType` 的含义、显示条件和取值来源，降低后续修改的理解成本：

```javascript
/**
 * 特殊汇总行类型说明（specialRowType）：
 *   subtotal   - 小计：前端累加所有普通费用行的 updBusCost
 *   freightDiff - 运费差价：serviceCharge（脱敏）；companyType !== 1 时显示
 *   insurance  - 货物保障服务：premiumAmount；premiumBuyerType === '10' 时显示
 *   oilDiscount - 油气优惠：-oilDiscountAmount；有油气配置时显示
 *   total      - 合计：小计 + 差价 + 保障 - 油气；companyType !== 1 时显示
 *   fpDiff     - 全流程运费差价：fpFreightDiffAmt；fullProcessFlag === 10 时显示
 *   shipperPay - 客户应付运费：payableFpFreight；全流程 companyType === 2 时显示
 *   carrierPay - 应付承运商运费：payableFpFreight（加粗）；全流程 companyType === 1 时显示
 */
```

## 四、跨模块共性问题（与 `waybillSettlement` 共享）

### 问题 11：`getHandlerList` 在两个模块里完全相同

```javascript
// driverChecking/index.vue 和 waybillSettlement/index.vue 里完全相同
async getHandlerList() {
  try {
    const companyId = this.$store.state.app.userInfo.companyId
    const data = await listAvailableEmployeeByCompanyId({ companyId })
    const list = data?.model || []
    this.handlerList = list
      .map(item => ({ label: item.employeeUserName || '', value: item.userBaseId }))
      .filter(item => item.value)
  } catch (err) {
    this.handlerList = []
  }
}
```

**优化方向**：提取到 `networkMainBodyMixin.js` 或单独的 `settlementCommonMixin.js`。

---

### 问题 12：`getNotice`（走马灯）在两个模块里完全相同

走马灯公告逻辑（接口调用 + 文字长度判断容器大小）两个文件各自维护，内容完全一样。

**优化方向**：提取到公共 mixin。

## 五、优先级与推进节奏

| 优先级 | 优化项 | 收益 |
|--------|--------|------|
| P0 | `filterModel` 格式化，每字段独占一行（问题 2） | 零风险，即时提升可读性 |
| P0 | 删除 `computed funcId` 里的 `console.log`（问题 8） | 零风险 |
| P0 | 提取静态枚举常量到 enums（问题 3） | 消灭跨模块重复定义 |
| P1 | 提取 `networkMainBodyMixin.js`（问题 4/11/12） | 消灭三个完全重复的方法 |
| P1 | `handleStar` 迁入 unitTool（问题 5） | 消灭多文件重复 |
| P1 | `buildTime` 迁入 utils/func.js（问题 6） | 规范工具函数归属 |
| P1 | `add.conf.js` / `add.format.js` 补充说明注释（问题 9/10） | 大幅降低后续维护成本 |
| P2 | 搜索区模板配置化（问题 1） | 收益最大，改动也最大 |
| P3 | `formatLongText` / `cutLongText` 归位（问题 7） | 代码归属规范化 |
