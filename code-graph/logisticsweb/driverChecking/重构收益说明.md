# 司机对账模块 — 现状问题与重构收益说明

> 目的：当前代码存在的真实风险和维护成本，以及重构后的预期收益

---

## 一句话总结

司机对账模块 7 个文件承载了列表筛选（22 个搜索字段 + 58 列表头）、费用核对、审核三大核心流程，存在 **2 个已确认 bug**（1 个影响数据展示准确性、1 个导致列表信息缺失），以及大量「看不懂就不敢改」的魔法数字和重复代码，每次新需求迭代时开发人员需要额外花 **30%~50%** 的时间理解现有逻辑才能动手。

---

## 二、现状问题（附真实场景）

### 问题 1：费用汇总行重复叠加，金额展示错误

**场景还原**：
运营打开一张运单的对账详情页，查看费用明细。此时页面正常——有一行小计、一行合计。运营切到其他浏览器 tab 处理别的事，再切回来，发现费用表格里出现了 **2 行小计、2 行合计**，金额翻倍展示。如果继续切换，行数还会继续增加。

**根因**：`$total()` 方法每次被调用时会往数组里追加汇总行，但只 `pop()` 掉最后一行。浏览器 tab 切换触发了 Vue 的 `activated` 生命周期重新调用该方法，旧的汇总行没有被清理干净，新的又追加上去。

**影响**：
- 运营看到的费用合计金额翻倍或多倍，可能据此做出错误的对账判断
- 如果在核对模式下发生，运营可能基于错误的小计去确认金额

**对应重构项**：T1（改动量：2 行代码——方法开头加一行 filter）

---

### 问题 2：列表页部分单元格显示空白，信息缺失

**场景还原**：
运营在司机对账列表页浏览数据，发现某些运单的「运输类型」「车辆类型」「货运类型」列显示为**空白**——不是"--"也不是"无"，就是什么都没有。运营无法判断这张运单是公路还是水路，只能点进详情页去确认。

实际上这些运单是有数据的，只是后端返回了一个格式化方法没覆盖到的 code 值。

**根因**：`codeFormat` 方法在 switch 没有命中时，`options` 变量未定义，`options?.find()` 返回 `undefined`，渲染到单元格就是空白。缺少一个兜底的 `return '--'`。

**影响**：
- 列表页信息不完整，运营需要逐条点进详情页确认
- 批量筛选时无法通过列表直接判断运输类型，降低处理效率

**对应重构项**：T2（改动量：1 行代码）

---

### 问题 3：代码里全是「魔法数字」，改错一个数字就是线上事故

**场景还原——日常开发**：

产品提了一个需求："批量确认按钮，把'审核通过'状态也加进去"。开发人员打开代码，看到：

```js
// index.vue — 批量确认方法
const validList = this.selection.filter(item =>
  ['110', '130', '200'].includes(item.busStatus)
)
```

开发人员需要：
1. 先查文档确认 `'110'` 是什么状态（待审核）、`'130'` 是什么（审核通过）、`'200'` 是什么（待对账）
2. 确认要加的新状态 code 是多少
3. 同时检查 `handleBatchDisPass` 里的 `['110']` 是否也要改
4. 搜索 `$action` 方法里 40 行 switch/case，确认新状态的按钮显示逻辑

如果手滑把 `'130'` 写成 `'1300'`，代码不会报错，测试时很难覆盖到全部状态组合，直到线上运营发现"审核通过的运单批量确认失败"才暴露。

**类似的裸数字散落在 4 个文件里**，包括：
- 费用类型 `'200'`/`'600'`（应付/其他应付）
- 货运类型 `'1'`/`'2'`（传统/网货）
- 操作模式 `'read'`/`'verify'`/`'check'`
- 8 种账单状态 `100`~`220`

**影响**：
- 新人上手需要反复查文档确认数字含义
- 改错一个数字 = 线上 bug，且不会有编译报错
- 无法通过 IDE 搜索"所有用到待审核状态的地方"

**对应重构项**：T9~T12（新建 constants.js + 逐文件引入）

### 问题 4：$total 方法 80 行混在一起，改一种行要通读全部

**场景还原**：

产品说："全流程运单的'油气优惠'行显示条件要调整，改成只有 oilAmount > 100 时才显示"。开发人员打开 `add.format.js`，找到 `$total` 方法——80 行代码，小计/运费差价/货物保障/油气优惠/合计/全流程差价/客户应付/承运商应付的构建逻辑全堆在一起。

开发人员需要：
1. 从第 1 行读到第 80 行，找到"油气优惠"的构建位置（大约在第 45 行附近）
2. 理解它和前后行的依赖关系（合计行的金额包含油气优惠吗？全流程差价行呢？）
3. 小心翼翼地修改条件，担心影响上下文的其他行

改完后，回归测试需要覆盖 **8 种特殊行 × 3 种 action × 全流程/非全流程 × B1/非B1** 的排列组合，因为所有逻辑耦合在一起，改任何一处都可能影响其他行。

**影响**：
- 简单的条件调整需要通读 80 行代码
- 回归范围大（所有特殊行都可能受影响）
- 新增一种特殊行时不确定该插在哪

**对应重构项**：T13（$total 拆为 6 个子方法）

---

### 问题 5：$action 40 行 switch/case，状态权限全貌看不清

**场景还原**：

产品问："'处理中'(115) 状态下，运营能做什么操作？"开发人员打开 `$action` 方法：

```js
$action(row, action) {
  switch (+row.busStatus) {
    case 100:
      if (action === 'read') return true
      if (action === 'verify') return true
      return false
    case 110:
      if (action === 'read') return true
      if (action === 'check') return true
      return false
    // ... 40 行类似代码
  }
}
```

需要从上往下逐个 case 找到 115，然后一行行看哪些 action 返回 true。如果要回答"哪些状态可以退回"，需要在 40 行里逐个搜索 `giveBack`。

更麻烦的是：如果要新增一个操作类型（比如"催办"），需要在 **8 个 case 里都加一行**，漏加一个就是 bug。

**影响**：
- 无法一眼看清"8 个状态 × 6 种操作"的完整映射
- 回答产品的权限问题需要逐行读代码
- 新增操作类型要改 8 处，容易遗漏

**对应重构项**：T14（改为一张二维映射表）

---

### 问题 6：两个完全相同的方法同时存在，改了一个忘改另一个

**场景还原**：

后端调整了附件接口的字段名，从 `attachmentName` 改为 `fileName`。开发人员搜索发现了 `formatTableFilse` 方法并修改了。但代码里还有一个 **完全相同实现** 的 `formatResultFilse` 方法（只是名字不同），忘了一起改。结果：费用表格的附件能正常展示，但审核结果的附件就挂了。

**影响**：
- 重复方法维护成本翻倍
- 改了一个漏了另一个 = 线上 bug

**对应重构项**：T6（合并为 1 个方法）

## 三、数字化对比

| 维度 | 现状 | 重构后 |
|------|------|--------|
| 已知 bug | 2 个（汇总行叠加 + 列表空白） | 0 |
| 魔法数字 | 散落 4 个文件，含义靠猜/查文档 | 集中 1 个文件，看常量名直接懂 |
| $total 方法 | 80 行混在一起，改任一行影响全部 | 主体 10 行 + 6 个独立子方法，互不影响 |
| $action 方法 | 40 行 switch/case，全貌不可见 | 1 张映射表，8×6 关系一目了然 |
| 重复方法 | formatTableFilse / formatResultFilse 完全相同 | 合并为 1 个，改一处即生效 |
| 费用表格模板 | 内联 add.vue（200 行），三种模式条件混写 | 独立组件，改列只改 1 个文件 |
| 新增一种特殊行 | 通读 80 行找插入点 + 回归全部行 | 写 1 个子方法 + 加 1 行调用 |
| 新增一种操作按钮 | 在 8 个 case 里各加 1 行（容易遗漏） | 映射表每行加 1 个字段 |
| 新增一种状态 | switch 里加 1 个 case + 所有批量方法确认 | 映射表加 1 行 + 常量文件加 1 行 |

## 四、风险说明（不做的代价）

### 短期风险（当前已存在）

| 风险 | 严重程度 | 说明 |
|------|:-------:|------|
| 费用汇总行叠加 | **高** | 金额翻倍展示，运营可能据此确认错误金额 |
| 列表空白单元格 | 中 | 信息缺失，运营需逐条点进详情确认 |

### 中期风险（随业务迭代加剧）

| 风险 | 说明 |
|------|------|
| 改一处坏一片 | 80 行混合逻辑，修改油气行可能影响合计行 |
| 魔法数字写错无感知 | 把 `'130'` 写成 `'1300'` 不会报错，直到线上才暴露 |
| 重复代码同步遗漏 | 改了一个方法忘了另一个 = 线上 bug |
| 新人上手慢 | 理解 $total 的 8 种特殊行逻辑需要半天以上 |
| 需求交付慢 | 简单的"加一种特殊行"需要通读 80 行 + 全量回归 |

### 长期风险

司机对账是高频使用模块（日均操作量大，月底结算高峰更是密集使用）。随着业务持续迭代（全流程运单、油气优惠、分次付款等新功能不断加入），当前的代码结构会让每次迭代的成本递增。如果不在可控阶段做渐进式优化，后续要么在某次大需求里被迫重写（风险和成本远大于现在），要么开发人员选择"加 if/else 绕过去"让代码越来越难维护。

## 五、重构策略（为什么风险可控）

| 策略 | 说明 |
|------|------|
| 渐进式，不一刀切 | 拆成 15 个独立单元，每个可单独测试、单独上线 |
| 先修 bug，再优化结构 | 第一波只修 2 个 bug（各 1~2 行改动），立刻有用户价值 |
| 不改业务逻辑 | 重构只调整代码组织方式，接口调用和业务规则完全不变 |
| 每步有明确验证清单 | 每个单元列出测试场景和边界情况，QA 可对照回归 |
| 可随时停止 | 做完任何一个单元都是稳定状态，不存在"做了一半必须做完" |
| 改动量极小 | 第一波 2 个 bug 各只改 1~2 行；第二波 6 项清理各只删几行废注释 |

## 六、收益总结

### 对用户/业务

- 消除费用汇总行叠加 bug → 运营看到的金额始终准确
- 消除列表空白单元格 → 信息展示完整，不需要逐条点进详情
- 代码结构清晰后，新需求（如新增特殊行、新增操作按钮）上线更快，等待时间缩短

### 对开发效率

- 新增一种特殊行：从"通读 80 行找位置 + 全量回归"变为"写一个子方法 + 加一行调用 + 只回归该行"
- 新增一种操作按钮：从"8 个 case 各加一行"变为"映射表每行加一个字段"
- 状态值含义：从"查文档/问同事"变为"看常量名直接懂"
- 新人理解模块：从"半天以上"变为"看映射表 + 子方法名就懂"

### 对代码质量

- 2 个已知 bug 清零
- 魔法数字全部常量化，改错会有 IDE 提示
- 重复代码合并，改一处即生效
- 核心方法从"80 行混合"变为"6 个独立子方法"，单独可测
